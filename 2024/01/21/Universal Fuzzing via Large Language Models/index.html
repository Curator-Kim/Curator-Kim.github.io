Universal Fuzzing via Large Language Models

Universal Fuzzing via Large Language Models (fuzz4all.github.io)

Abstract: 介绍Fuzz4ALL——通用模糊器，可以针对多种不同的输入语言和这些语言的多种不同特征进行测试，利用大型语言模型（LLM）作为输入生成和突变引擎。提出了一种新颖的自动提示技术，用于创建适合模糊测试的LLM提示，和一种由LLM驱动的模糊测试循环，用于迭代更新提示来创建新的模糊测试输入。

使用六种不同语言（C、C++、Go、SMT2、Java 和 Python）作为输入的九个待测系统上对 Fuzz4All 进行了评估，评估结果表明，通用模糊测试的覆盖率高于现有的特定语言模糊测试工具，此外，Fuzz4All 还在 GCC、Clang、Z3、CVC5、OpenJDK 和 Qiskit 量子计算平台等广泛使用的系统中发现了 76 个漏洞，其中 47 个漏洞已被开发人员确认为以前未知的。

3 FUZZ4ALL Approach

图一概述了论文的方法，首先接收任意用户输入，这些输入描述了模数测试要产生的输出，如SUT的文档、示例代码片段或规范。由于用户输入可能冗长、多余且不相关，蒸馏LLM会将用户输入蒸馏为简洁而信息丰富的模糊测试提示词。蒸馏LLM会对多个不同的候选信息进行采样，从而执行自动提示步骤(3.1节)。每个候选提示词都会传递给生成 LLM，以生成代码片段（即模糊输入）。然后，Fuzz4All 会选择产生最高质量模糊输入的提示 。

由图也可以看到，Fuzz4All基于两个模型，一个是浓缩用户输入的蒸馏LLM，另一个是创建模糊测试输入的生成LLM，来平衡不同LLM所提供的成本和收益之间的权衡。蒸馏LLM使用了具有强大自然语言理解能力的高端大型基础模型，生成LLM使用一个较小的模型作为生成 LLM，使用最先进的 GPT4 和 StarCoder实现了 Fuzz4All。(因为蒸馏LLM需要理解和蒸馏任意用户输入，而自回归推理的成本较高，生成LLM需要高效率)

使用通过自动提示选择的最佳提示作为生成 LLM 的初始输入提示，进入到模糊测试循环阶段中(3.2 节)，Fuzz4ALL会在循环中不断对生成LLM进行采样来生成模糊测试输入，为了避免生成许多类似的模糊测试输入，Fuzz4ALL会在每次迭代中不断更新输入提示。具体来说，该方法会选择一个之前生成的输入作为示例，示例展示了我们希望模型生成的未来输入类型。除了示例之外，Fuzz4All 还会在初始提示中附加一个生成指令，引导模型生成新的模糊输入。这一过程不断重复，同时不断将生成的模糊输入传递到SUT中，并检查是否触发崩溃。

3.1 Autoprompting

Autoprompting将用户输入提炼为适合fuzzing的提示词。与传统fuzzer需要标准格式的输入不同，FUZZ4ALL可以直接理解自然语言形容和用户输入的代码例子。由于用户输入可能是不相关、冗余的，直接使用用户输入作为生成LLM的提示词可能低效，这一点在第 5.3 节的消融研究中得到了证实。因此，自动提示的目标是生成经过提炼的输入提示，以实现有效的基于 LLM 的模糊处理。 #### 3.1.1 Autoprompting Algorithm 算法1详细解释了FUZZ4ALL的Autoprompting步骤，算法的输入是用户输入以及要生成的提示词数量，输出是用于fuzz的输入提示词。由于目标是使用提炼 LLM 生成提示，对用户提供的信息进行提炼，因此向蒸馏LLM发出以下指令：“请以简洁的方式总结上述信息，描述目标的用法和功能”。 令$M_D$为蒸馏LLM，*UserInput*为用户输入，*APInstruction*为自动提示指令，生成的提示*prompt*可以形式化为条件概率 $$ M_D(prompt|userInput,APInstruction) $$ Fuzz4ALL首先使用0度的贪婪采样算法生成候选提示词。通过先进行低温采样，算法可以获取可信度较高的合理解决方案。这个方法也常用于其他领域，例如程序合成，首先对贪婪输出进行评估，检查它是否能解决问题。然后算法会提高采样温度，来获得更多样化的提示。与贪婪算法相比，高温采样算法能得到不同的提示，每个提示都能对用户输入进行独特的提炼总结。每个生成的提示都会添加到候选提示列表中（第 6 行），直到算法达到所需的候选提示数量。 为了选出最佳提示词，算法会进行小规模fuzz实验来评估每个提示词。更具体来说，该方法将每个提示作为生成 LLM 的输入，为每个提示生成多个代码片段。然后，Fuzz4All 根据评分函数对每个提示生成的代码片段进行评分，这里的评分标准是有效的唯一生成代码片段的数量，即被目标 SUT 接受的代码片段的数量，因为希望模糊输入有效或接近有效，以触发 SUT 内部的逻辑。假设$M_G$是生成LLM，p是候选提示词，isValid是一个函数，当生成代码有效时返回1，否则返回0。默认得分函数定义为$\sum_{c\in M_G(p)}[isVaild(c,SUT)]$。最终FUZZ4ALL会选择得分最高的输入提示词作为fuzz的初始输入提示词。总之，自动提示步骤结合了提示生成和评分功能，使 Fuzz4All 能够自动生成/选择适合模糊目标的提示。 #### 3.1.2 Example: Autoprompting.

图2展示了Autoprompting算法产生的一个输入提示词例子：fuzz c++23的新特性std::expected。用户输入将原始的cppreference文档传递给Fuzz4ALL，有多个表格和冗长的提示(498词，3262字)。相反，autoprompting算法提供了创造的蒸馏输入提供了目标特性更为精确的自然语言描述(214词，1410字)，包含对如何使用 std::expected 的高级描述。使用提炼后的输入提示，Fuzz4All 可以生成有效针对 C++ 编译器的 std::expected 功能的模糊输入。

3.1.3 Comparison with Existing Autoprompting Techniques.

第一个在软件工程任务中使用黑盒autoPrompting从用户输入自动化蒸馏知识。相比之前NLP和软件工程的autoPrompting通过访问模型梯度来优化提示，论文的autoprompting只需要对蒸馏LLM进行黑盒采样访问。尽管使用评分函数来评价prompt类似于NLP，但论文的评分函数直接评估生成有效代码片段的确切下游任务的提示，而不是使用近似的代理评分函数。

3.2 Fuzzing Loop

Fuzzing Loop的目标就是使用生成LLM产生多样化的fuzzing输入。然而，由于 LLM 的概率性质，使用相同输入进行多次采样会产生相同或相似的代码片段。为生成多样化的输入，利用LLMs的能力，即利用例子和自然语言指令来指导生成。

使用例子的目标是指明生成LLM要产生的代码片段类型。生成策略被设计为如何处理所提供代码示例的指令。在模糊循环的每次新迭代之前，Fuzz4All 都会在输入提示中添加一个示例和一个生成策略，从而使生成 LLM 能够不断创建新的模糊输入。

3.2.1 Fuzzing Loop Algorithm

算法2描述了Fuzzing Loop，初始输入为初始提示词和时间预算，最终输出为bugs。首先，算法初始化生成策略（generate-new、mutate-existing 和 semantic-equiv），这些策略将用于在模糊循环过程中修改输入提示，图三列出了三种生成策略和对应的指令。在第一次调用生成 LLM（用 $M_G$ 表示）时，算法还没有任何模糊测试输入的示例。因此，算法会在输入提示中添加generate-new生成指令，引导模型生成第一批模糊输入（第 3 行）。

然后，算法进入主fuzzing-loop，不断更新提示来创建新的模糊测试输入。算法会从上一批生成的模糊输入中选择一个示例，从所有对 SUT 有效的模糊输入中随机挑选。除示例外，算法还会从三种生成策略中随机选择一种。算法将初始输入提示、所选示例和所选生成策略合并为一个新的提示，然后用该提示查询生成 LLM，以生成另一批模糊输入

主循环重复进行直到预算耗尽。对于每个创建的模糊输入，Fuzz4All 都会将输入传递给 SUT。如果检测到意外行为如崩溃，算法会将报告添加到集合中。

3.2.2 Example: Fuzzing Loop

图三展示了fuzzing loop如何生成模糊测试输入。例子中正在fuzz一个SMT Solver，其要求输入SMT2语言表示的逻辑表达式。

开始时没有例子，因此使用generate-new策略来生成fuzz输入

然后，算法请求模型根据产生的输入，进行mutate-existing策略，对选定的输入进行变异。新的模糊测试输入会通过交换变量类型、增加计算量来修改之前的输入

在下一个fuzz回合中，算法会选择semantic-equiv生成策略，改变之前的输入而不改变其语义。一般通过简单加一个语法tag给选定的样例。

4.1 Implementation

FUZZ4ALL主要用Python实现，其中autoprompting和fuzzing loop组件仅包括872LoC。对比传统fuzzer，例如CSMITH(>80K LoC)，需要大量人力来实现Generator，FUZZ4ALL非常轻量级。

FUZZ4ALL使用了GPT4作为蒸馏LLM，通过OpenAI的API提供，max_token为500。

FUZZ4ALL使用了StarCoder模型的Hugging Face实现作为生成LLM，

默认设置：temperature: 1, batch size: 30, 最大输出长度为 1,024，使用核采样，top-p 为 1。

5 RESULTS

5.1 RQ1: Comparison against Existing Fuzzers

5.1.1 Coverage over Time

图 4 显示了 Fuzz4All 与基线相比的 24 小时覆盖率趋势，其中实线表示平均覆盖率，区域表示五次运行中的最小值和最大值。我们发现，在模糊测试结束时，Fuzz4All 的覆盖率在所有目标中最高，与性能最好的基线相比，平均提高了 36.8%。与基于生成的模糊器（即 YARPGen 和 MorphQ）相比，Fuzz4All 几乎能立即实现更高的覆盖率，这表明与传统的程序生成技术相比，LLM 在生成多样化代码片段方面具有强大的生成能力。虽然基于突变的模糊器（即 go-fuzz 和 GrayC）在开始时能够通过使用高质量的种子获得更高的覆盖率，但通过突变获得的覆盖率会迅速下降，而 Fuzz4All 则能够缓慢但稳定地覆盖更多代码。请注意，为了进行公平比较，我们将自动提示时间作为模糊测试预算的一部分，它产生的开销可以忽略不计（平均每次模糊测试 2.3 分钟）。

基线模糊器在 24 小时结束时会达到覆盖高原，而 Fuzz4All 则与之不同，即使在模糊活动接近尾声时，它仍能不断找到覆盖新代码的输入。回想一下，在Fuzz4All模糊循环的每次迭代中，原始输入提示都会更新为新示例和生成策略（第3.2节），促使LLM生成新的模糊输入。我们假设，即使经过长时间的模糊处理，Fuzz4All 也能有效地生成新的、多样化的模糊输入，从而持续提高覆盖率。

5.1.2 Generation Validity, Number, and Coverage

我们检查了所研究的 SUT 中生成的模糊输入的数量及其有效率。在表 2 中，”# 程序 “一栏表示生成的唯一输入的数量，”有效百分比 “表示有效模糊输入的百分比，”覆盖率 “表示每个模糊器最终获得的覆盖率以及与最佳基线相比的相对改进。我们首先发现，几乎所有传统模糊工具都能达到很高的有效率，只有 Hephaestus 除外，因为它故意生成无效代码（主要是错误类型），以检查编译错误。相比之下，Fuzz4All 生成的有效模糊输入比例较低（与基准工具相比，平均降低了 56.0%）。此外，基线工具生成的模糊输入的原始数量也要高得多。通过使用 LLM 作为生成引擎，Fuzz4All 受到了 GPU 推理的瓶颈，导致模糊输入比传统模糊器少了 43.0%。

尽管有效率和模糊输入的数量较低，但与传统模糊工具相比，Fuzz4All 生成的程序更加多样化，这一点可以从获得的高覆盖率（平均增加 36.8%）得到证明。此外，即使是接近有效的无效代码片段也可以用于模糊测试，因为它们可以发现 SUT 验证逻辑中的错误。在第 5.4 节中，我们将进一步介绍 Fuzz4All 利用有效和无效代码片段检测到的各类错误，以进一步展示生成不同模糊输入的好处。

我们注意到，Fuzz4All 在不同的 SUT 中实现了广泛的有效率和模糊输入数。由于每次模糊迭代后调用 SUT 进行错误检测的成本不同，因此不同目标的模糊输入数量也不同。关于有效率，通用编程语言（如 C 语言）与特定领域语言（如用于 SMT 解算器的 SMT2 语言）相比，有效率相对较低。更严格的语言，如 Go，不允许任何已声明但未使用的变量，其有效率甚至更低。我们还发现，对量子计算平台进行模糊测试的有效率也很低。由于量子计算是一个新兴领域，有自己的一套应用程序接口库，因此一代 LLM 在训练过程中看到的量子程序示例可能不如更成熟的语言多。尽管如此，Fuzz4All 仍能利用用户提供的文档生成有趣的模糊输入，这些输入利用了量子库 API，与最先进的模糊器相比，覆盖率有了显著提高（+75.6%）。

5.2 RQ2: Effectiveness of Targeted Fuzzing

评估 Fuzz4All 执行有针对性模糊测试的能力，即生成针对特定特征的模糊输入的能力。对于每种目标 SUT 和语言，都会针对三种不同的示例特征进行测试，并将它们与 RQ1（第 4.3 节所述）中使用的一般用户输入设置进行比较。这些特征包括内置库或函数/API（Go、C++ 和 Qiskit）、语言关键字（C 和 Java）以及理论（SMT）。目标模糊运行的用户输入是我们关注的特定功能的文档。表 3 显示了定向模糊测试的结果以及 RQ1 中使用的默认通用模糊测试的结果。每一列代表一次有针对性的模糊测试运行，我们重点关注一个特征。每个单元格中的值显示了特定模糊运行的特征命中率（第 4.3 节）。我们还包含了所获得的覆盖结果。

我们发现，针对特定特征的模糊输入直接使用了该特征，平均命中率高达 83.0%。这一结果表明，Fuzz4All 通过向生成的 LLM 发送描述特定特征的输入提示，确实可以进行有针对性的模糊处理。此外，我们还观察到，对相关的特征进行模糊处理会导致中等程度的跨特征命中率（即特征 X 在特征 Y 的模糊运行中的命中率）。例如，C 关键字 typedef 和 union 都与类型操作有关，因此，与不相关的特征（如 goto）相比，它们的交叉特征命中率较高。如表 3 所示，通用模糊方法虽然能实现最高的整体代码覆盖率，但在针对特定特征进行模糊时却效率极低（与 Fuzz4All 的针对性模糊相比，命中率平均降低了 96.0%）。例如，在 Qiskit 中，一般模糊测试对三个目标特征的命中率为 0%。这是因为这些特征是最近才添加到 Qiskit 中的，尚未被广泛使用，因此在 LLM 训练数据中极为罕见。不过，通过在目标模糊活动中提供适当的用户输入，Fuzz4All 可以成功生成使用这些新特征的模糊输入。Fuzz4All 的这种能力对于想要测试 SUT 的新功能或组件的开发人员来说非常有价值。

5.3 RQ3: Ablation Study

为了研究 Fuzz4All 的每个组件如何对整体模糊效果做出贡献，我们根据 Fuzz4All 的两个关键组件进行了消融研究：（a）自动提示，即提供给生成 LLM 的初始输入提示类型；（b）模糊循环，即使用选定的示例和生成策略。我们研究了这两个关键部分的三种变体。表 4 显示了所研究变体的覆盖率和有效率。

5.3.1 Autoprompting

首先，我们研究了为生成 LLM 提供的不同初始输入的影响。为了减少其他因素的影响，我们将生成策略固定为只使用 generate-new，并研究了三种变体1：1）无输入：不使用任何初始提示；2）原始提示：直接使用原始用户输入作为初始提示；3）自动提示：使用自动提示生成初始提示。我们发现，在所有研究语言中，无输入变量的覆盖率最低。在 “无输入 “中，我们不提供任何初始提示，而初始提示提供了关于我们要生成模糊输入的特征的有用信息。因此，LLM 只能生成具有高有效率的简单代码片段，但在覆盖 SUT 方面却不太有效。在使用原始提示变体时，我们观察到覆盖率有所提高，在这种变体中，我们提供原始文档作为初始提示。不过，我们可以通过自动提示阶段将用户输入提炼为简洁但信息丰富的提示（自动提示），而不是使用原始用户输入，从而进一步提高代码覆盖率和有效率。直接使用用户提供的输入可能会包含与模糊测试无关的信息，从而导致验证率降低（因为生成的 LLM 可能难以理解原始文档）和覆盖率降低（因为与我们的自动提示生成的提示不同，原始文档并非设计用于生成 LLM）。

5.3.2 Fuzzing Loop

接下来，我们通过保持初始提示相同（使用默认的自动提示）来研究模糊循环设置的不同变体： 1）无示例：在模糊循环过程中不选择示例（即持续从相同的初始提示中采样）；2）有示例：选择示例，但只使用新生成指令2；3）Fuzz4All：使用所有生成策略的完整方法。我们首先观察到，只从相同的输入（不含示例）中采样，LLM 通常会重复生成相同或相似的模糊输入。平均而言，有8.0%的模糊输入在 “无示例 “情况下重复生成，而在使用完整的Fuzz4All方法时，这一比例仅为4.7%。在输入提示中添加示例（w/example）可避免从相同分布中采样，从而提高覆盖率和有效率。最后，在所有 SUT 中，完全 Fuzz4All 方法的覆盖率最高。与 w/ example 变体（第二好）相比，full Fuzz4All 增加了额外的生成策略：semantic-equiv 和 mutate-existing，这有助于进一步为生成 LLM 提供有用的指令。

5.4 RQ4: Bugfinding

表 5 总结了 Fuzz4All 在我们研究的九个 SUT 上发现的错误。Fuzz4All 共检测到 76 个错误，其中 47 个错误已被开发人员确认为以前未知的错误。这些结果不仅证明了 Fuzz4All 在发现大量错误方面的实际效果，还证明了 Fuzz4All 在跨语言和跨 SUT 方面的通用性。

5.4.1 Examples

图 5a 显示了在 GCC 中使用 noexcept(x) 时发现的一个 bug，这是 C++ 的一个特性，它规定如果 x 值为 true，函数就不会抛出。在这个错误示例中，Fuzz4All 使用 std::optional 生成了一段相当复杂的代码，表示运行时可能存在也可能不存在某个特定值。虽然这段代码是有效的，而且应该能正确编译，但这种难以处理的运行时依赖关系组合会导致 GCC 崩溃，出现内部编译器错误。我们注意到，由于先前的技术根本不支持 noexcept 功能，因此无法发现这一错误。有趣的是，他们甚至在 GCC 的官方测试套件中添加了我们提交的代码片段的略微修改版本。

图 5b 显示了在 Clang 中发现的一个错误，其中的无效代码导致了分段故障。Fuzz4All 使用了一种不同寻常的函数声明语法（即 auto x (…) -> return_type ），它利用了 C++ 中的 decltype 操作。然而，当首先评估 decltype 内的 throw 语句时，由于 throw 会提前退出作用域并导致 Clang 崩溃，因此会跳过对返回类型的评估，从而出现错误。这段代码虽然无效，但经开发人员确认，仍然有助于揭示 Clang 前端的一个错误。此外，以前的模糊工具很难发现这个错误，因为它们通常只关注生成有效代码，而不处理特别难以建模的 decltype 函数。

图 5c 显示了在 Go 中发现的一个 bug，即输入为 nil 会导致分段故障，而不是产生有用的故障信息。这个 bug 是通过运行 Go 标准库发现的，我们在该库中提供了文档，其中包括 ReadMemStats 函数的说明。开发人员已确认并修复了该错误。虽然这个漏洞看起来很简单（调用一个单一函数），但 go-fuzz 基线却无法发现它，原因很简单，因为 go-fuzz 需要手动编写模板来针对特定库，而运行时库并不属于任何此类模板的一部分。有了 Fuzz4All，用户只需提供相关输入信息（如文档），就能直接针对任何 Go 标准库。

图 5d 显示了在 Qiskit 的 QASM 输出程序中发现的一个错误。由 qc 变量表示的量子程序被导出到 QASM（一种低级表示法）时，会无声地生成一个无效的输出文件，导致重新导入时程序崩溃。问题在于，导出程序在 QASM 中使用寄存器的名称作为标识符，即 “crz”，这也是 QASM 语言中一个众所周知的操作名称，因此生成的代码含糊不清。需要注意的是，之前的研究 [56] 发现不了这个漏洞，因为他们使用的是预定义模板，只包含匿名寄存器，而 Fuzz4All 则有效地利用了 LLM 的量子知识，注入了一个有意义的字符串字面，从而检测到了这个漏洞。

6 Threats To Validity

内部威胁：主要来自Fuzz4All的实现。为此进行了代码审查和测试，以确保正确性

外部威胁：生成LLM利用的是在去年完成的训练中获得的知识，因此可能会因数据转移而降低未来的有效性。Fuzz4All可以通过自动提示步骤来缓解这一问题，在该步骤中，最新的文档/示例代码可以让模型生成最新的模糊输入。另外一个威胁来自使用蒸馏 LLM 生成初始输入，LLM 可能会产生 “幻觉”，即生成虚构或不准确的信息，大多数使用 LLM 的管道都存在这种局限性。
