Large language model are Zero-Shot Fuzzers: Fuzzing Deep-Learning Libraries via Large Language Models

Abstract: 传统的Fuzz技术对深度学习系统难以奏效，因为输入DL程序中需要同时满足输入语言（如 Python）的语法/语义和用于张量计算的 DL API 输入/形状约束。为解决这些局限性，提出了Titanfuzz，直接利用大型语言模型（LLM）生成用于模糊 DL 库的输入程序：现代 LLM 还可以在其训练语料库中包含大量调用 DL 库 API 的代码片段，因此可以隐式学习语言语法/语义和复杂的 DL API 约束，从而生成有效的 DL 程序。更具体地说，使用生成式和填充式 LLM（如 Codex/InCoder）来生成和变异用于模糊测试的有效/多样化输入 DL 程序。

实验结果表明，与 TensorFlow/PyTorch 上最先进的模糊器相比，TitanFuzz 的代码覆盖率分别提高了 30.38% 和 50.84%。此外，TitanFuzz 还能检测到 65 个错误，其中 41 个已被确认为以前未知的错误。

1 INTRODUCTION

此前对DL库的Fuzz工作主要分为两类：API-level fuzzing和model-level fuzzing：API 级模糊主要通过为每个目标 API 生成各种不同的输入来测试单个库的 API，从而发现潜在的崩溃或结果不一致。另一方面，模型级模糊技术旨在生成各种完整的 DL 模型，然后在不同的后端（如 Keras 的不同底层库 [36]）上比较模型输出，以发现潜在的错误。虽然模型级模糊技术和 API 级模糊技术在发现错误方面都很有效，但它们仍然存在以下局限性：

  1. 缺乏多样化的 API 序列。只孤立地fuzz单一API，无法揭示由连锁应用程序接口序列引起的错误。虽然模型级模糊技术[27, 55, 70]可以对 API 序列进行潜在测试，但其突变规则通常有严格的限制，覆盖能力有限。

    图 1a 显示了 PyTorch 中通过 API 序列暴露的一个错误示例。

  2. 不能产生任意代码。现有的 DL 库模糊器使用预定义的生成语法，侧重于更改程序的一小部分，限制了代码的多样性

    freefuzz和muffin的两个例子

    在模糊循环过程中，FreeFuzz 会对这些有效输入进行小规模突变以生成新的输入，例如更改数据类型（如将 float32 改为 float16）。因此，FreeFuzz 受限于跟踪参数空间和预定义的突变规则。与此同时，最先进的模型级模糊器 Muffin [25]，通过为每个操作的 API 和预定义的代码结构（如由顺序层组成的模型）使用手动注释的规范来生成各种模型，以保持模型的有效性。因此，在使用 DL 库应用程序接口时，这种先前的 DL 库模糊器无法充分探索现有的巨大搜索空间。

    图 1b 显示了 PyTorch 中的一个错误示例：clamp并没有在CPU上把-0改为+0，由于现有的应用程序接口级和模型级技术的生成方法受到限制，先前的工作漏掉了这个错误。

论文的工作：提出了TITANFUZZ——第一种通过大型预训练语言模型（LLM）对 DL 库进行模糊处理的全自动方法。泰坦 LLM 可以在其训练语料库中包含大量各种 DL 库的代码片段（例如，GitHub 上有 >400,000 个 TensorFlow/PyTorch 项目）使它们能够隐式地学习 Python 语法/语义和 DL API 的复杂类型/约束，从而直接生成/突变有效的 DL 程序，用于模糊 DL 库。

在 TitanFuzz 中，使用带有分步输入提示的生成式 LLM ，生成用于模糊处理的初始种子程序。采用进化策略，通过使用 LLM 自动突变种子程序来生成新的测试程序，进一步丰富测试程序池。为了生成更复杂、更多样的 API 调用关系，论文设计了一个适配函数，该函数根据数据依赖深度和独特库 API 的数量，对种子程序或突变测试程序进行优先排序，使我们能够发现只有在研究复杂 API 关系时才能发现的错误。最后，在不同的后端执行生成的测试程序并进行差异测试，以检测错误。事实上，图 1 中的两个错误都是以往任何 DL 库模糊器都无法检测到的，但 TitanFuzz 却检测到了这两个错误，并被开发人员确认为以前未知的错误。技术建立在 Codex 和 InCoder 上，因为它们分别在生成任务和填充任务上取得了最先进的结果。对 TensorFlow 和 PyTorch 进行了评估

总之，本文有以下贡献：

  • Dimension. opens a new dimension for fuzzing DL libraries. 并且方法可以轻松扩展到其他应用领域（如编译器、解释器、数据库系统、SMT 解算器和其他流行库）
  • Technique. 实现了能应用于DL库的Titanfuzz，使用生成式 LLM（Codex）合成高质量的种子输入，然后将填充式 LLM（InCoder）与进化算法相结合，引导生成更多独特的库 API 使用和有效/多样化的 DL 程序。
  • Study. 对Pytorch和Tensorflow进行了评估。结果表明，TitanFuzz能够覆盖1329/2215个API，在PyTorch和TensorFlow上的覆盖率分别为20.98%/39.97%，与最先进的模糊工具相比，API覆盖率提高了24.09%/91.11%，代码覆盖率提高了50.84%/30.38%。此外，TitanFuzz 还能检测出 65 个错误，其中 41 个已确认为以前未知的错误。此外，我们还进行了广泛的消融研究，以证明 TitanFuzz 中组件设计的合理性。

3 APPROACH

图四展示了TITANFUZZ的概览。

对于任何目标 API，TitanFuzz 都会首先使用生成式 LLM 生成用于模糊测试的高质量种子程序列表（第 3.1 节）。具体做法是为模型提供逐步提示[59]，生成直接使用目标 API 的代码片段。

对于生成的种子程序，进一步应用进化模糊算法迭代生成新的代码片段（第 3.2 节）。在每次迭代中，首先从种子库中选择一个具有高适应度得分的种子程序，然后使用不同的突变算子（第 3.2.1 节）系统地将所选种子程序的部分内容替换为masked-tokens，从而生成Masked-input。突变算子采用多臂老虎机算法[66]（第 3.2.2 节）进行选择，从而最大限度地产生有效且唯一的突变。利用掩码输入，利用填充 LLM 的能力进行代码填充，生成新代码来替换掩码标记（第 3.2.3 节）。

对于突变后的代码，首先过滤掉所有执行失败的代码，然后使用我们的适应度函数（第 3.2.4 节）对每个突变代码进行评分，然后将所有的突变代码放入种子库。在未来的突变中，优先选择得分较高的种子，从而生成更多样化的高质量代码片段用于模糊测试。

最后，使用不同的后端(GPU和CPU)来执行生成的代码，识别潜在的错误。

使用 Codex [12] 和 InCoder [21] 分别作为生成式和填充式模型。

3.2 Evolutionary Fuzzing

算法1描述了Titanfuzz的进化算法：首先，我们用 Codex 生成的种子初始化种子库（第 2 行）。种子库保存着迄今为止生成的代码片段列表。接下来，初始化每个突变算子的先验分布，并在选择突变算子的主循环中使用和更新（第 3 行）。然后进入生成循环，根据适合度得分选择当前的突变种子（第 5 行）。

决定在种子上应用哪种突变算子（第 6 行）。由于能帮助模型生成有效和独特突变的突变算子对于不同的目标 API 可能是不同的，因此我们使用多臂匪徒（MAB）[66] 算法动态学习算子优先级策略。每个突变运算符都会用特殊标记屏蔽掉当前种子程序的一个或多个片段（第 7 行）。然后将屏蔽输入输入到 InCoder 模型中，生成代码片段，对屏蔽区域进行采样填充（第 8 行）。对于生成的每个样本，我们都会运行并静态分析代码片段（第 9 行）。具体来说，我们要确定可以编译的代码片段（ValidSamples）。然后，我们根据产生的有效样本和无效样本的数量更新突变算子的后验分布（第 10 行）。对于每个有效样本，我们都会根据定义的适配度函数（FitnessFunction）计算适配度得分，该函数旨在优先处理不同 API 之间具有大量独特交互的各种种子，从而使我们能够发现更多潜在错误。我们会使用适配度得分将这些样本添加到种子库中，以便进行下一次迭代种子选择（第 12 行）。

3.2.1 Mutation Operators

使用了四种基础变异算子：argument, prefix, suffix, method。图 6 显示了使用每种突变算子生成的屏蔽输入示例。

首先确定种子代码片段中的目标 API（如 torch.mm）。每个突变运算符都会用掩码标记\替换所选种子代码的特定位置。例如，argument操作符将用 span 标记替换目标 API 调用的参数。关键是利用LLM补全SPAN掩去的代码

每一种突变算子类型：

  • Argument：将 API 调用参数替换为 span 标记，从而令填充模型可以向API中填入不同的参数。注意，参数替换突变算子并不局限于目标应用程序接口，它可以应用于代码片段中的任意库应用程序接口。此外还有 keyword-insertion突变算子，能够让模型为特定库 API 生成额外的关键字(关键字可以揭示许多有趣的错误)。

  • Prefix/Suffix：prefix/suffix算子选择目标API前/后的一个代码片段，插入SPAN标记。由于Prefix和Suffix会影响API的参数，一般将prefix/suffix和argument变异算子组合使用。

  • Method：用 span 标记替换随机选择的库 API 方法名称。这样做的目的是在使用现有参数的同时，生成不同的库 API 调用。

    在 DL 库中，相关 API 共享相同的输入是很常见的，借用一个 API 的输入可能会触发其相关 API 中的错误。我们还可以利用方法运算符生成更多独特的代码片段，测试更多的应用程序接口。

3.2.2 Mutation Operator Selection

将变异算子的选择问题表述为多臂老虎机问题。论文假设：对于不同的 DL API，效果好的突变算子（例如，产生更多触发错误的突变体）可能是不同的。因此，希望在生成循环中自适应地学习每个算子的有效性与优先性。

将突变算子选择问题建模为一个伯努利强盗问题：

Definition 1 . Bernoulli Bandit. 假设有K个臂，每个臂的结果要么成功要么失败。对于每条臂$i \in \{1,\dots,K\}$有相应的成功概率$\mu_i\in[0,1]$，而agent对此并不知晓。在每个时间步骤 𝑡，agent将拉动手臂 $𝑖_𝑡$ 并观察从伯努利分布 𝐵𝑒𝑟($𝜇_{𝑖𝑡}$ ) 中抽取的成功/失败输出。我们的目标是在𝑇轮实验中最大化累计成功次数。

3.2.4 Fitness Function

论文设计了一个适配函数来对生成的代码进行排序。论文应用静态分析来计算每个测试程序的适配度得分。生成的突变程序具有更深的执行路径、更多样的计算图和更复杂的 API 调用，则得分更高。

  • Depth of dataflow graph. 通过静态分析生成代码片段中变量的数据流来构建数据流图，每条边代表两个操作之间的数据依赖关系。数据流图的深度 (D) 定义为图中任意路径的最大边数。
  • Number of API calls. 计算每个代码片段中存在的唯一库 API 调用 (U) 的数量。由于 LLM 往往会生成许多代码行重复的代码片段，因此也计算并惩罚同一输入中重复调用的库 API 的数量(R)

给定生成的代码片段C，定义适配函数

根据计算公式，TitanFuzz 会优先选择涉及长链 API 序列和更多独特 API 的输入程序。这样，我们就能覆盖不同 API 之间的更多交互，从而可能触发更多有趣的程序行为/错误。与此同时，如果只使用前两个子项，TitanFuzz 就会覆盖越来越长的 API 序列和重复的 API 调用，从而降低模糊处理的效率。因此，TitanFuzz 会进一步惩罚重复调用 API 的序列。

3.3 Oracle

利用不同的测试oracle来找bug，主要聚焦于以下bug类型

Wrong-Computation. 比较两个执行后端所有中间变量的值，并在值有显著差异时发现错误计算。由于某些计算的非确定性会导致 CPU 或 GPU 上的结果略有不同，因此我们沿用了之前的研究成果 [71]，使用容差阈值来检查数值是否存在显著差异。计算值的差异可能表明库应用程序接口的不同后端实现或不同应用程序接口之间的交互存在潜在的语义错误。

Crash. 在执行过程中可能会遇到意外崩溃，例如分段故障、中止、INTERNAL_ASSERT_FAILED 错误
