模糊测试技术综述_仁泽众等

模糊测试技术是漏洞检测技术的一种，通过使用针对目标程序生成的随机字符流，队目标程序进行多次测试，以检测可能存在的漏洞

1 模糊测试

1.1模糊测试的发展历程

1.2 模糊测试基本工作流程

基本上划分为五步：预处理(preprossessing)、输入构造(input buiding)、输入选择(input selection)、评估(evaluation)、结果分析(post-fuzzing)。其中第一和第五个环节属于模糊测试开始前的准备工作和模糊测试结束后的收尾工作，实际测试是第2~4个环节

  • 预处理环节：搜集目标相关信息并制定模糊测试的策略，为监控目标在测试中的运行状态做必要的准备，通常依赖于插桩、符号执行以及污点分析这类程序分析技术
  • 输入构造：首先获取一定数量的种子，随后确定种子的能量分配策略、种子的优先级以及种子的突变策略，最后依据这些信息获得大量的输入数据
  • 输入选择：同股票提前过滤掉无效的输入数据，以节省模糊测试的时间，该环节适合使用机器学习技术，通过完成模式识别任务，在输入被执行前就提前判定数据的有效性
  • 评估：设计合适的实验，依照评估指标队模糊测试进行评估。一个合适的模糊测试实验通常需要考虑：选择合适的对象作为模糊测试的目标、选择合适的重复测试次数以及测试使用的超时时间等
  • 结果分析：对获得的测试结果进行去重、复现、分析以及威胁性评估等工作，最终确定是否发现了有价值的漏洞

2 预处理

2.1 分析技术与获取的信息

1）插桩

插桩技术通过向目标的代码中合适的位置添加预设好的代码，获得程序的静态或动态执行信息，比如程序的抽象语法树，覆盖率以及函数内变量取值等

常用的插桩技术分为动态插桩和动态插桩2种。静态插桩通常在源码或者中间代码的编译过程中进行，常见的比如通过GCC编译器在汇编语言上插桩，LLVM在生成的中间语言LLVM IR(low level virtual machine intermediate representation)上插桩．优点是节省时间、速度快，缺点是依赖于程序源码．动态插桩则是在运行的过程中对运行过的代码进行插桩．比如利用QEMU等模拟技术，进行动态插桩，可以获得程序运行时的信息，缺点是资源的开销大。插桩技术已经被应用到模糊测试中，比如AFL中就有静态插桩和动态插桩的２种模式，使得AFL可以在源码以及二进制代码上进行模糊测试．插桩技术的缺点是会带来资源的开销，以AFL为例的灰盒测试引入了将近1300％的开销．

2）符号执行

符号执行将程序行为的推理归结为逻辑领域的推理，通过构建一个表示程序执行的逻辑公式，可以同时推断一个程序在不同输入上的行为。该方法可以使模糊测试获得较好的覆盖率，并可以深入到程序深处，探寻可能存在的漏洞区域。

符号执行可分为静态符号执行以及动态符号执行

静态符号执行通常会因为程序中循环和递归的存在，陷入到路径爆炸中，还会因为路径约束中包含如取Hash值等操作，导致约束求解失败

动态符号执行使用较多，通常执行通过对程序进行实际执行和符号化执行，维护程序的实际状态和符号化状态，维护程序的实际状态和符号化状态，通过将难以求解的约束替换为实际值，缓解了静态符号执行的问题，并按照深度优先的搜索策略对目标程序进行了探索

动态符号执行也有自己的问题

  1. 程序分支存在，导致路径爆炸的问题仍然存在，一种解决方法是通过启发式的方法，选择比较重要的路径进行探索
  2. 使用实际值替换的方法会丢失一些路径，造成探索结果的不完整
  3. 如何处理类似取余操作这样的非线性约束

3）污点分析

该技术会观测程序中，哪些程序数据受到了预先准备好的污染源（比如输入）的污染，目的是跟踪污染源和汇聚点（比如有敏感信息的程序数据）之间的信息流。分为静态污点分析和动态污点分析

静态污点分析不需要程序实际运行，通过对程序静态分析，获得程序控制流图，抽象语法树等信息，依据数据流以及依赖关系进行污点分析；

动态污点分析则是在程序实际执行的过程中，利用程序的动态执行信息进行污点分析。

重要研究方向：减少污点分析跟踪的对象、降低污点分析的开销、提升模糊测试的检测效率

可以获得充足的目标内部信息

3 实际测试环节

该部分由输入构造、输入选择、评估三部分组成

3.1 输入构造

输入构造环节的目的是构造出可以被检测目标执行，用于模糊测试的输入数据，具体的挑战是如何在尽量满足语法语义检查的情况下，短时间内生成大量的输入，用于对目标做全面而深入分析

为获得足够多的高质量输入数据，需要经过种子获取、种子筛选、种子突变3个阶段

3.1.1 种子获取

  • 提前准备好的高质量数据集
  • 模型生成
  • 从执行过的数据进行选择

3.1.2 种子筛选

获得备选种子后，需要对种子池里的备选种子进行筛选划分不同权重，以确定每个种子要用来生成多少输入，按照什么样的顺序从种子池中选择种子

1）种子的能量分配

一个种子蕴含的能量，代表了该种子可以生成输入数据的多少。关注能量分配策略，直接影响模糊测试的执行速度以及覆盖率

  • AFLFast将模糊测试过程建模为马尔科夫链状态空间的遍历过程，是能量分配更合理
  • EcoFuzz使用多臂老虎机模型建模模糊测试的过程，通过使用基于平均损耗的自适应能量分配策略，有效减少了能量损耗
  • AFLGo使用基于模拟退火算法的能量调度方法，更偏向于距离目标更近的路径
  • Hawkeye全面搜集与目标程序位置相关的调用图、函数以及基本块层面的距离信息
2）种子的优先级

决定了模糊测试从种子池中选择种子的顺序

AFL中引入了进化算法，通过适应度函数可以对输入进行评估，并从中选取最佳的输入作为种子放入种子池，该种子池实际是一个队列，种子按照入队顺序依次用于模糊测试。

长度越短的种子，占用内存越少，加快测试的速度，因此会获得更高的优先级，AFL采取了在维持覆盖率不变的条件下，通过反复删除种子的不同部分，来试图缩短种子的大小

3.1.3 种子突变

变异策略太保守——覆盖率偏低；变异策略太激进——生成大量无效的输入，影响效率

突变的具体操作有6种：

  1. 比特翻转
  2. 简单算数运算：按照8b的步长，依次按照8b,16b,32b的宽度，从头开始进行加减操作
  3. 覆盖：使用预设的值将种子中的一部分进行覆盖，或者将其插入到种子的一些位置
  4. 插入：使用预设的值插入到种子的一些位置
  5. 删除：删除种子的某些部分
  6. 拼接：选择两个差异较大的种子进行拼接

Path sensitive fuzzing for Native Application

1 Introduction

代码覆盖率对于覆盖引导的模糊器至关重要

由于高额的开销使得跟踪路径覆盖不可行

1.1 覆盖不精确如何影响漏洞发现

  1. 覆盖不精确可能会导致测试器没有能够区分两条不同的路径。如果一个测试用例探索到了一条新路径，但他与之前已探索过的路径碰撞了，测试器则会认为这条新路径不那么重要
  2. 覆盖不精确会影响测试策略。它影响力测试器选择最优的种子进行测试

1.2 如何提高覆盖精确度并引导测试器

跟踪边缘和块覆盖是可能的

AFL的覆盖跟踪有着更低的时间开销

由于哈希冲突问题，AFL的边缘覆盖是不完美的。对该问题的解决措施是扩大AFL用于存储覆盖的比特图的大小

CollAFL解决了AFL的哈希碰撞问题并提高了覆盖精确度，还提出了三个新的测试策略，来驱动测试器发现未探索的道路，并提高发现漏洞的效率

CollAFL通过确认每条路径有一个特定的Hash，所以AFL能区分每一条路径

1.3 如何将建议的解决方案应用于二进制应用程序？

使用静态二进制检查(static binary instrumentation(SBI))来将解决方案应用到二进制程序中。对于每一个二进制程序，fuzzer会反汇编该程序，恢复其控制流图，并使用建议的覆盖跟踪方案对应用程序进行检测

1.4 覆盖敏感的测试器表现如何

测试结果

  • 在真实应用中哈希碰撞问题很普遍，75%边界会与其他产生冲突
  • 论文中所提出的碰撞减轻方案可以解决所有已知边界的哈希碰撞，并能够帮助测试器探索9.9%多的代码，250%多独特的冲突和140%多的安全漏洞

3 Imporve Coverage Accuracy

3.1 覆盖粒度

三种覆盖粒度：块覆盖、边覆盖、路径覆盖

典型的块覆盖方案会跟踪测试中每个块的击中次数。然而它不跟踪块的顺序，导致了覆盖信息的损失

典型的边覆盖跟踪每条边的击中次数，然而它同样不跟踪边的次序，同样导致了信息的损失。

路径覆盖方案会跟踪边的顺序，提供最完整的代码覆盖信息。然而代码路径非常长，且数量庞大，时间与空间开销难以承受，实际上不可行。一种优化方法是上下文敏感的边覆盖，能够跟踪部分函数调用顺序。这种方法有较高的性能开销和低效率。

边覆盖方案达到了性能的平衡。然而存在哈希碰撞的问题。CollAFL采用边覆盖跟踪策略解决了这一问题。

3.2 哈希冲突的简单解决方案

直接解决方案是扩大哈希空间，即AFL中的位图大小，然而当前默认的位图大小(64KB)已经是一种性能上的折中

这种方法会带来性能上的大量损失。要减少5%的哈希碰撞，需要将位图大小从4KB扩大到4MB，导致60%执行速度上的损失。

3.5 实现细节

3.5.1 CollAFL for Applications With Source Code

CollAFL扩展了AFL的llvm模式，并写了Clang链路时间优化传递来

  1. 获取所需的基本块和边信息
  2. 给每个块分配密钥
  3. 根据每个基本块的类型解析哈希计算算法
  4. 用哈希计算和覆盖跟踪代码检测每个块

在第一步，使用CLang默认实施获取后例和先例信息，例如。通过API llvm::TerminatorInst::getSuccessor.然而，离线解决间接控制转移的目标是一个公开的挑战，这会影响我们所需的先例信息的准确性。例如，它可能错误地将一些基本块分类为单个（或没有）先例块。

因此采取两步来改良。第一，将不被任何人直接调用的函数的入口块标记为多先例块。另外，将间接调用指令展开为一组直接调用和一个间接调用指令，类似于去虚拟化技术。因此能够连接基本块在一起，减少单先例的块，能够更多地使用Fmul而不是Fsingle来去计算块的哈希，减少碰撞的可能性。

(不懂：为啥要减少单先例的块，单先例的算法开销不是更小吗？)

3.5.2 CollAFL-Bin for Binary Applications

选择使用静态二进制检测来应用到边碰撞减少策略。

CollAFL-bin的综合检测过程如下：第一，使用API openBinary来加载二进制程序，并调用API getImage来获取程序镜像。然后通过API getModules获取所有模块，通过API getProcedures识别初始执行模块并获取所有函数。第二，遍历每个函数的所有基本块后获得CFG。有了CFG，就如有源码的CollAFL执行，即在没有冲突的情况下搜索相关的哈希id，并为每个基本块搜索各自的实施模式。最后，为三种实施模式创建调用函数。

静态分析中还有许多问题，间接跳转指令导致的不准确的控制流图….

4 Prioritize Seed Selection

直接驱动fuzzer到未探索的道路

  • 如果一条道路上上有很多未探索的邻近分支，那么这条道路的变异很可能能探索这些未探索的分支
  • 如果一条道路上上有很多未探索的邻近后代，那么这条道路的变异很可能能探索这些未探索的后代

提升漏洞发现的效率

  • 如果一条道路上有许多内存存储操作，很有可能能够触发潜在的内存破坏漏洞，变异策略同样。

基于这些策略，提出以下种子选择策略

4.1 Untouched-Neighbour-Branch Guided Policy

能够探索更多为探索过的邻近分支的种子将会被赋予更高的优先级。将该策略称为CollAFL-br

将未探索的邻近边的数量成为测试例T的权重，函数IsUntouched返回1当且仅当边没有被之前任何测试例覆盖

随着测试的进行，测试集也会发生变化，因此一个测试例的权重是动态的

4.2 Untouched-Neighbour-Descendant Guided Policy

具有更多未探索的邻近后代会更有更高优先级，这种策略会具有更高优先级。这种策略称为CollAFL-desc

将未探索的邻近后代数量称为测试例T的权重

记为CollAFL-mem，有更多内存读取操作的种子会具有更高优先级

4.4 Implementation Details

在运行时，首先检查测试用例的覆盖位图，并得到这个测试用例所覆盖的所有边和命中数。由于每个边都有不同的哈希值，我们可以从每个边的哈希值解码出它的开始和结束块。然后，对于每个块，我们将根据整体覆盖位图，得到其未触及的邻居分支的列表。加上已收集到的子路径和内存访问操作数，可以相应计算出三种策略的权重。

5.4 Vulnerablities

如表六所示，在200小时的检测中，发现了超过7500个独特的冲突，并确认了157个漏洞

冲突分类分析。首先，利用开源工具afl-collect来消除重复的冲突。(它主要使用GDB插件exploitable)来计算每个崩溃backtrace的哈希值，根据哈希值来去除重复的崩溃。然后使用AddressSanitizer重新变异目标程序，使用去重后的冲突样本进行测试，AddressSanitizer随后可以报告这些崩溃的根本原因，并进一步去重。最后进行手动分析，确认了157个独特的漏洞。

值得注意的是，由于性能开销，我们只利用了AddressSanitizer来分类漏洞，并没有将它用在CollAFL，因此CollAFL有较好的执行速度，但却可能错失潜在的不会引起崩溃的漏洞。
