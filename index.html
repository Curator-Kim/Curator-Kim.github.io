ICS3Fuzzer: A Framework for Discovering Protocol Implementation Bugs in ICS Supervisory Software by Fuzzing

对ICS系统中监控软件(supervisory software)进行fuzz

方法：构建了监控软件执行轨迹与对应输入的state-book，利用状态选择算法分配更多的资源给Interesting states。而通过在模糊测试期间同步管理外部时间来达到interesting states。

1 INTRODUCTION

目前已有的工作主要有三个问题

  1. 依赖程序源代码
  2. 只能操控服务端的程序
  3. 在协议通信的时候不能同步GUI管理
  4. 一些工作能够处理私有协议，但并不够综合，不能应用于fuzz监控软件

有两种方案可用于部署所提出的模糊测试方法：

  • 让专用的 PLC 设备进行模糊测试；
  • 模拟 PLC 设备的响应。

经作者测试，第一种方案速度慢，而且无法扩展，因此提出了一种混合解决方案，既利用了真实 PLC 和模拟响应的优势，又避免了它们的劣势。

总而言之，作者做出了以下贡献

  • 设计并实现了 ICS3Fuzzer，可用于fuzz ICS监控软件，并且可移植，定制化
  • 提出了新的fuzz策略，根据执行轨迹和相应的输入来选择输入状态。
  • 使用 ICS3Fuzzer 测试了来自不同供应商的 4 种不同的商业监控软件。发现了 13 个零日漏洞，并获得了 3 个 CVE，其中 2 个被列为关键漏洞

3 DESIGN OVERVIEW

3.1 Movitating Example

如图为一tcp通信过程

key observation.

  1. TCP会话的起点通常是人工操作员通过监控软件提供的图形用户界面执行的按键操作。只要操作员点击相应的按钮，监控软件就会与 PLC 设备进行三方握手。然后双向交换数据包。
  2. 在会话过程中，在许多情况下操作员会执行额外的按键操作。例如，在向 PLC 下载程序时，操作员首先点击 “连接 “按钮，创建一个新的 TCP 会话。
  3. 最后，会话通过定期交换心跳信息或停止所有信息交换而结束。

New Insight.

在不同的交互状态下，监控软件通常会有不同的行为（即代码执行）。监控软件的交互状态主要是指与 PLC 设备交互的特定程序执行环境。交互状态由多个因素决定，包括之前的按键操作和之前的输入状态。例如，如果操作员没有首先按下 “连接 “按钮，”下载 “按钮就会保持灰色，无法点击。再比如，能否测试输入状态取决于当前的交互状态是否正确。这也表明，监控软件的协议实现与图形用户界面高度耦合。

3.2 Challenges

  1. 引导监控软件进入到特定的输入状态

    • 所有TCP会话由监控软件客户端发起，Fuzzer需要被动的等待客户端的会话发起请求
    • 要模糊一个输入状态，我们必须一次又一次地触发监控软件进入正确的交互状态，这涉及到三个实体之间的同步：按下图形用户界面操作按钮、监控软件中的代码执行和设备响应
    • 由于输入状态之间存在复杂的依赖关系，在许多情况下，如果监管软件不首先输入其他输入状态，就无法直接输入一个输入状态。更糟糕的是，根据流量确定一套相当完整的输入状态本身就是一个具有挑战性的问题。请注意，不同的监控软件有不同的输入状态。

  2. 未知数据帧格式和状态空间的私有协议

  3. 模拟私有协议的会话

    在进入特定的输入状态之前，监督软件发出的每一个请求，我们都必须做出正确的回应。一种简单直接的方法是使用真实设备来维持交互。然而，这种解决方案依赖于昂贵的硬件，而且不具备可扩展性。

    模拟会话是一个很有前途的选择，但需要对专有协议有全面的了解。

Solution:

  • 对于挑战1，作者设计了一种新的控制机制，通过对图形用户界面操作和网络通信的精确同步控制，实现进入任何确定的输入状态。
  • 对于挑战2，作者对协议的数据包帧格式进行逆向工程。还进行了差分分析，以确定会话 ID、序列号等字段并识别其值约束。为了识别有价值的协议状态并过滤重复的状态，避免对详细的程序状态进行逆向工程。相反，我们根据执行跟踪和相应的输入来构建state book。为了实现良好的状态覆盖并提高 ICS3Fuzzer 的效率，作者提出了一种动态切换状态的新模糊策略。
  • 对于挑战3，作者根据实际捕获的流量构建了一套通信模板/模式，以此模拟 PLC 设备的响应。对于来自监控软件的每个请求，ICS3Fuzzer 首先会识别匹配模板中的相关响应。然后，ICS3Fuzzer 会自动查找/调整每个相关字段的相应值约束（即会话 ID、序列号等）（如有）。如果无法找到过于复杂的约束和依赖关系，则只能求助于真实的 PLC 设备

4 DETAILED DESIGN

ICS3Fuzzer设计如下图所示。主要包含两个阶段：

  1. 预处理阶段，ICS3Fuzzer产生必要信息和工具促使模糊测试阶段自动化
  2. 模糊测试阶段，ICS3fuzzer动态切换输入阶段，并输入产生的测试样例和观察监控软件的状态

在预处理阶段，分析给定监控软件的功能和专有协议。

在功能性方面，关注如何在模糊循环中自动启动会话。

在专有协议分析方面，重点关注如何获取其状态空间/格式的重要知识，以帮助选择有价值的输入状态并生成有效的突变数据。

此外，还关注如何根据捕获的流量模拟通信，以避免受到专用硬件的限制。

在模糊测试阶段，该阶段为全自动的，由四个步骤组成协调工作（见图 5 右侧）。

首先，它根据state book提供的信息选择有价值的输入状态。

其次，它根据推断出的协议格式知识生成变异输入。

然后，在自动同步管理图形用户界面操作和网络流量的基础上，将变异数据送入选定的输入状态。

最后，它监控监控软件的状态，并记录畸形输入以备重现。

4.2.2 Obtaining State-Space.

该步骤有助于识别和区分会话中的输入状态。

从监控软件的角度来看，激活的会话涉及许多输入状态（例如，激活 “Proficy Machine Edition “的 “显示状态 “功能时，会话中有 100 多个输入状态）。但是，也有许多输入状态是重复的、相似的（例如，与接收心跳信息相对应的状态）。

要区分这些输入状态，应该记录在相应输入状态下触发的执行轨迹，因为即使信息差异很小，但执行轨迹可能截然不同，也很容易区分。

因此，在进行模糊测试时，能够触发丰富而不同的执行轨迹的输入状态更值得关注。有了这些知识，就可以切换有价值的输入状态，从而更有效地进行模糊测试（见 4.3）。

作者根据代码执行路径和相应的输入构建状态簿，而不恢复输入状态的特定语义。为了获取代码执行路径，利用 DynamoRIO对目标监控软件进行动态检测。只记录/转储了信息传输过程中执行的基本块的位图（通过挂钩和跟踪 “send()/recv() “函数）。因为只对输入状态下的信息处理逻辑感兴趣。

同时，记录原始信息及其命令。因此，state-book 中的每个输入状态都可以用元组表示，包括原始消息、执行跟踪和索引。

如果无法解决动态字段问题，ICS3Fuzzer将求助于真实的 PLC 设备。具体来说，对于监控软件发出的每一个请求，ICS3Fuzzer 都会将其转发给真实的 PLC 设备，并分析设备的响应，然后决定是将响应变异还是直接转发给监控软件。因此，在进入特定的输入状态之前，监控软件发出的任何请求，ICS3Fuzzer 都能做出正确的响应。

4.3 State Selection

根据构建的state-book，每个输入状态有三个属性：index, execution trace, original message。

作者为每个属性分配一个数值，这样每个状态都有一个权重，通过利用这些权重来选择最有希望模糊的状态。权重分配基于三个假设。

  • 网络通信越 “深入”，就越有可能存在漏洞；
  • 当信息转发到监控软件时，触发的新的基本模块越多，就越有可能存在漏洞；
  • 输入越复杂，就越有可能导致崩溃。

注意，在建立state book时，会忽略与现有状态具有高度相似性的状态，例如那些表现出相同重复行为的状态，避免将深度 “高 “但实际上根据协议规范非常肤浅的状态（例如，有许多心跳信息的状态）放入状态簿中。

作者使用 Jaccard 指数来定量衡量消息的相似性，即两个消息序列之间共同字段的数量除以两个消息序列联合体中总字段的数量。

位图相似度的简单计算方法是位图中的公共位数除以位图的总命中数。最终的相似度是信息相似度和位图相似度相乘的结果。

通过设置阙值，过滤掉过于相似的状态。阙值在实验中被设置为80%

4.6 Crash Monitor

监控基于 Windows 事件日志服务。一旦应用程序崩溃，Windows 系统的事件日志中就会添加一条记录，标记为 “应用程序错误”。该记录不仅包含崩溃信息，还提供了崩溃的上下文以协助诊断。因此，在每个模糊周期中，我们都会检查事件日志中应用程序类别的实时记录。

目前ICS3Fuzzer不考虑挂起作为漏洞的指标，因为检测 ICS 操作延迟并将其作为错误指标会产生大量误报。

目前还没有标准来指定区分正常响应和延迟响应的临界值，因此很难过滤掉误报。

5.3 Performance

由于模糊测试的非确定性，在每个测试周期之后，监控软件的图形用户界面也是非确定的。例如，图形用户界面可能被冻结，或者会出现一些不可预见的干扰。

为了确保模糊处理过程的稳定性，也为了实施的方便，ICS3Fuzz在输入测试用例并检查程序的状态后，会杀死该进程并重新启动软件进行新的回合。因此，时间成本主要由四部分组成：重新启动监控软件、操作图形用户界面、网络通信和其他。

表 5 给出了实验的平均时间成本。单个测试用例的速度较慢（5-20 秒一个测试用例），但它可以完全自动化，并且并行化处理

Path sensitive fuzzing for Native Application

1 Introduction

代码覆盖率对于覆盖引导的模糊器至关重要

由于高额的开销使得跟踪路径覆盖不可行

1.1 覆盖不精确如何影响漏洞发现

  1. 覆盖不精确可能会导致测试器没有能够区分两条不同的路径。如果一个测试用例探索到了一条新路径，但他与之前已探索过的路径碰撞了，测试器则会认为这条新路径不那么重要
  2. 覆盖不精确会影响测试策略。它影响力测试器选择最优的种子进行测试

1.2 如何提高覆盖精确度并引导测试器

跟踪边缘和块覆盖是可能的

AFL的覆盖跟踪有着更低的时间开销

由于哈希冲突问题，AFL的边缘覆盖是不完美的。对该问题的解决措施是扩大AFL用于存储覆盖的比特图的大小

CollAFL解决了AFL的哈希碰撞问题并提高了覆盖精确度，还提出了三个新的测试策略，来驱动测试器发现未探索的道路，并提高发现漏洞的效率

CollAFL通过确认每条路径有一个特定的Hash，所以AFL能区分每一条路径

1.3 如何将建议的解决方案应用于二进制应用程序？

使用静态二进制检查(static binary instrumentation(SBI))来将解决方案应用到二进制程序中。对于每一个二进制程序，fuzzer会反汇编该程序，恢复其控制流图，并使用建议的覆盖跟踪方案对应用程序进行检测

1.4 覆盖敏感的测试器表现如何

测试结果

  • 在真实应用中哈希碰撞问题很普遍，75%边界会与其他产生冲突
  • 论文中所提出的碰撞减轻方案可以解决所有已知边界的哈希碰撞，并能够帮助测试器探索9.9%多的代码，250%多独特的冲突和140%多的安全漏洞
  • 在碰撞减轻方案和未接触分支指向种子选择的联合作用下，探索多了20%的代码，320%多的冲突，260%多的安全漏洞

总结：

  • 哈希碰撞问题严重影响了AFL的效率，覆盖不精确的问题降低了覆盖引导的fuzzer的效率
  • 设计了一套算法来解决AFL中的哈希碰撞问题，低开销情况下提高了边覆盖精确度
  • 提出了三个覆盖敏感的种子选择策略。经验表明基于精确的边覆盖信息来权衡种子选择能提高fuzzer的表现
  • 提供了CollAFL的两个变种，能够在提供或不提供源码的情况下测试程序

3 Imporve Coverage Accuracy

3.1 覆盖粒度

三种覆盖粒度：块覆盖、边覆盖、路径覆盖

典型的块覆盖方案会跟踪测试中每个块的击中次数。然而它不跟踪块的顺序，导致了覆盖信息的损失

典型的边覆盖跟踪每条边的击中次数，然而它同样不跟踪边的次序，同样导致了信息的损失。

路径覆盖方案会跟踪边的顺序，提供最完整的代码覆盖信息。然而代码路径非常长，且数量庞大，时间与空间开销难以承受，实际上不可行。一种优化方法是上下文敏感的边覆盖，能够跟踪部分函数调用顺序。这种方法有较高的性能开销和低效率。

边覆盖方案达到了性能的平衡。然而存在哈希碰撞的问题。CollAFL采用边覆盖跟踪策略解决了这一问题。

3.2 哈希冲突的简单解决方案

直接解决方案是扩大哈希空间，即AFL中的位图大小，然而当前默认的位图大小(64KB)已经是一种性能上的折中

这种方法会带来性能上的大量损失。要减少5%的哈希碰撞，需要将位图大小从4KB扩大到4MB，导致60%执行速度上的损失。

3.5 实现细节

3.5.1 CollAFL for Applications With Source Code

CollAFL扩展了AFL的llvm模式，并写了Clang链路时间优化传递来

  1. 获取所需的基本块和边信息
  2. 给每个块分配密钥
  3. 根据每个基本块的类型解析哈希计算算法
  4. 用哈希计算和覆盖跟踪代码检测每个块

在第一步，使用CLang默认实施获取后例和先例信息，例如。通过API llvm::TerminatorInst::getSuccessor.然而，离线解决间接控制转移的目标是一个公开的挑战，这会影响我们所需的先例信息的准确性。例如，它可能错误地将一些基本块分类为单个（或没有）先例块。

因此采取两步来改良。第一，将不被任何人直接调用的函数的入口块标记为多先例块。另外，将间接调用指令展开为一组直接调用和一个间接调用指令，类似于去虚拟化技术。因此能够连接基本块在一起，减少单先例的块，能够更多地使用Fmul而不是Fsingle来去计算块的哈希，减少碰撞的可能性。

(不懂：为啥要减少单先例的块，单先例的算法开销不是更小吗？)

3.5.2 CollAFL-Bin for Binary Applications

选择使用静态二进制检测来应用到边碰撞减少策略。

CollAFL-bin的综合检测过程如下：第一，使用API openBinary来加载二进制程序，并调用API getImage来获取程序镜像。然后通过API getModules获取所有模块，通过API getProcedures识别初始执行模块并获取所有函数。第二，遍历每个函数的所有基本块后获得CFG。有了CFG，就如有源码的CollAFL执行，即在没有冲突的情况下搜索相关的哈希id，并为每个基本块搜索各自的实施模式。最后，为三种实施模式创建调用函数。

4 Prioritize Seed Selection

直接驱动fuzzer到未探索的道路

  • 如果一条道路上上有很多未探索的邻近分支，那么这条道路的变异很可能能探索这些未探索的分支
  • 如果一条道路上上有很多未探索的邻近后代，那么这条道路的变异很可能能探索这些未探索的后代

提升漏洞发现的效率

  • 如果一条道路上有许多内存存储操作，很有可能能够触发潜在的内存破坏漏洞，变异策略同样。

基于这些策略，提出以下种子选择策略

4.1 Untouched-Neighbour-Branch Guided Policy

能够探索更多为探索过的邻近分支的种子将会被赋予更高的优先级。将该策略称为CollAFL-br

将未探索的邻近边的数量成为测试例T的权重，函数IsUntouched返回1当且仅当边没有被之前任何测试例覆盖

随着测试的进行，测试集也会发生变化，因此一个测试例的权重是动态的

4.2 Untouched-Neighbour-Descendant Guided Policy

具有更多未探索的邻近后代会更有更高优先级，这种策略会具有更高优先级。这种策略称为CollAFL-desc

将未探索的邻近后代数量称为测试例T的权重

4.4 Implementation Details

在运行时，首先检查测试用例的覆盖位图，并得到这个测试用例所覆盖的所有边和命中数。由于每个边都有不同的哈希值，我们可以从每个边的哈希值解码出它的开始和结束块。然后，对于每个块，我们将根据整体覆盖位图，得到其未触及的邻居分支的列表。加上已收集到的子路径和内存访问操作数，可以相应计算出三种策略的权重。
