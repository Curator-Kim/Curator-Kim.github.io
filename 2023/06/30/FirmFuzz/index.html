Firm-Fuzz

Firm-Fuzz

Firm-Fuzz：基于Linux硬件镜像的设备无关性仿真和动态分析框架

Introduction

基于Linux的嵌入式硬件主要有三处漏洞利用：

  • Linux内核的各个变种
  • 开源软件包
  • 供应商软件

主要关注：vendor-developed applications 供应商开发的软件

三个挑战：

  • 只接受符合语法的输入
  • 需要finegrained monitoring of the runtime environment 分析和监控运行环境
  • 需要硬件运行时环境

克服挑战：

  • 利用嵌入式设备的web应用接口作为入口点，产生合法输入
  • 注入运行时监控器到嵌入式设备的运行时环境中
  • 模拟硬件镜像，达到硬件无关性

先前的工作已经研究了在闭源硬件上的大规模仿真。这些工作将分析拓展到了许多镜像，但对于发现特定漏洞而言，这些太过通用，在不联系嵌入式系统和观察系统行为的情况下可能会错过大量漏洞

通过模拟固件，我们可以调整运行时环境，并在执行期间内省运行中的系统，观察漏洞扫描程序与系统的交互。这反过来又使我们能够发现深层次的脆弱性。

与传统的web应用程序扫描仪不同，FirmFuzz利用模拟固件提供的自由度来增强漏洞检测过程。它将运行时监视器集成到固件文件系统中，修改固件本身以改进错误查找过程。使用我们的语法合法输入生成策略和运行时监视器，我们在每个固件的基础上定制我们的分析，使我们能够触发固件中的深层错误。

总而言之，三个贡献

  • FirmFuzz 源码：https://github.com/HexHive/FirmFuzz，自动化仿真和动态分析框架
  • 开发了一个Fuzzer，能够产生合乎语法的输入，同时监控固件运行时
  • 测试了一些硬件镜像，发现了7个漏洞

Firmware Preprossing

FirmFuzz通过QEMU模拟层实现，工作流程如下所示

2.1 Information Gathering Phase

该阶段有两个目标：

  • 发现通过认证的用户名——密码对，从而提高覆盖率
  • 静态分析，从而发现PHP中漏洞代码的输入路径

基于PHP的应用程序中的CI漏洞是由于未初始化的用户输入被传递给不安全的PHP函数（如system或shell_exec）而产生的。FirmFuzz执行PHP脚本的程序内静态污染分析。它污染用户控制的变量$_GET、$_POST，并记录污染流向不安全函数的代码路径。对于这些代码路径中的每一个，都会构建一个约束集。FirmFuzz使用此集合来生成可能触发易受攻击代码路径的输入。

2.2 Firmware Preparation

该阶段创造一个镜像来进行仿真。通过使用外围设备映射策略创建固件，从而近似运行时环境。FirmFuzz 利用完整系统模拟，并将helper二进制文件注入到文件系统中，并增强内核从而以较低的漏报率/误报率发现漏洞。模拟器的后端允许硬件程序和漏洞查找工具进行交互。

2.2.1 Peripheral Mapping 外围映射

在缺少外围设备时，硬件可能无法正常工作。

在硬件需要一个外围设备时，FirmFuzz会提供一个到外围设备的伪映射，在硬件询问外围设备时，总是返回True。但由于IoT设备的多样性，这种方法并不总是稳定的

固件映像在自定义内核的模拟器下运行。内核被配置为在访问不受支持的设备时死机。FirmFuzz使用此崩溃日志创建设备到伪设备驱动程序的映射。FirmFuzz反复执行这个过程，直到所有不受支持的设备都映射到一个假设备。

2.2.2 Helper Injection.

Helper二进制文件允许FirmFuzz在模拟过程中检查固件。helper程序在固件的运行时环境中运行，并且在当前实现中检测CI漏洞。在模糊处理期间，如果执行此helper文件，FirmFuzz会标记CI漏洞。这种方法是完全自动化的，与固件无关；即，不依赖于固件上存在的实用程序来检测漏洞。

为了检测BO漏洞，论文使用了类似于Firmadyne的Linux内核的异常处理机制。FirmFuzz进一步利用该机制来检测NPD漏洞。FirmFuzz和Firmadyne在检测BO方面的主要区别在于，Firmadynne的作者在特定Netgear路由器服务的网页中手动发现了BO漏洞，他们通过向该特定网页发出curl请求来触发该漏洞，从而验证了该漏洞。然而，FirmFuzz不需要人工分析来检测BO漏洞。使用自定义模糊驱动程序，它会自动触发模拟固件中的BO漏洞，并打包PoC漏洞以重新创建该漏洞。对于FirmFuzz针对的XSS漏洞，我们观察到主机端监控足以进行检测

2.2.3 Network Configuration

硬件镜像在如何命名和分配地址给LAN/WAN接口时有所不同。这里采用和Firmadyne相同的方法推断这些网络配置：在“网络推理”模式下运行仿真，Firmfuzz记录固件和内核网络接口的所有交互。使用这些日志，FirmFuzz推断网络配置，并创建适当的虚拟网络接口，以允许与仿真固件进行交互。

Firmware Fuzzing

FirmFuzz通过增强模糊目标的仿真环境来帮助发现漏洞，从而改变了仿真固件中的漏洞检测模式。论文的方法消除了对服务器反馈的依赖，并允许在现场直接观察触发的漏洞。

FirmFuzz模糊器的三个主要特征：

  • 上下文驱动的输入生成-它结合了固件在与攻击面的不同部分交互时提供的上下文信息
  • 确定性漏洞检测-在客户机（即模拟固件）和主机中运行的漏洞监视器允许确定性漏洞检测
  • 如果固件在被模糊化时达到不一致的状态，FirmFuzz会自动将固件恢复到稳定状态。这允许对仿真目标进行连续模糊处理，而不需要手动干预来重置固件状态

由固件设置的web应用程序通过使用客户端JavaScript代码和服务器端代码的组合来提供其功能。FirmFuzz使用由我们的fuzzer控制的无头浏览器，通过web应用程序与固件交互，执行客户端JavaScript代码。使用来自这些交互的上下文信息，它生成模糊输入。模糊器和仿真固件之间的所有网络流量都通过代理服务器。这允许代理服务器捕获将由FirmFuzz变异的候选输入。

FirmFuzz在模糊化阶段针对的四种漏洞类型是：CI、BO、NPD和XSS。为了检测这些漏洞，FirmFuzz首先与web应用程序交互，生成一个合法的HTTP请求作为种子输入，以模糊测试表面的一部分，参考第3.1节。然后，它根据目标漏洞使用有效载荷对合法请求进行变异，并将其发送到固件。在发送变异请求时，FirmFuzz会监控固件以检测漏洞。如果检测到漏洞，变异请求将作为PoC输入与接收方URL一起记录，以允许再现

3.1 Syntactically Legal Input Generation

需要创建格式良好的HTTP请求:Firmfuzz必须了解模拟固件的Web页面设置，才能成功交互，但不同厂商的web页面设置各有差别。

在评估FirmFuzz的过程中，论文基于所测试的镜像，创建了所遇到的web界面的交互模板。但这种接口支持需要分析员手动验证。这种验证是将分析扩展到其他设备的限制因素。

3.2 Deterministic Bug Discovery

ZAP等扫描仪自动化扫描漏洞(如CI, BO 和NPD)的方法依赖于服务端的响应，但这种方法并不精确，可能会错误地定位漏洞，如BO，BPD，甚至会错失漏洞，也可能由于不准确的服务端响应误检。

为了提高检测准确性，FirmFuzz修改了固件的仿真环境和固件本身

FirmFuzz通过在执行测试输入时监控增强固件生成的日志来检测CI、BO和NPD漏洞。对于CI，它监视execve系统调用以记录毒二进制文件的执行（请参见第2.2.2节）。对于BO和NPD，它监视内核日志以查看是否有固件进程试图访问未映射的内存。

检测XSS漏洞不需要任何客户端协助；FirmFuzz的主机端监控足以检测到它，类似于现有的漏洞扫描程序。这种监控的形式是检测作为请求中的有效负载发送的JavaScript代码片段是否在机器中执行

3.3 Elimination of Fuzzing Side-Effects

Fuzz副作用：FirmFuzz在Fuzz过程中需要频繁与仿真固件交互，因此可能进入无法响应的状态。不一致的状态包括固件试图重新启动以应用某些更改，或者由于触发的功能被错误地模拟而导致无限循环。在这种状态下，固件变得对模糊没有响应，并且需要回滚到稳定状态。

3.4 Payload Delivery

需要来自Web段的合法请求，能够应用于变异并绕过有效性检测来发送给硬件

Fuzzer需要一组种子输入和可被接受的有效URL。由于web应用程序通常由JavaScript的host组件和server组件构成，需要驱动host组件触发所有可能的主机路径的生成。FirmFuzz利用无头浏览器从固件加载网页，然后遍历DOM，通过迭代网页上的所有按钮元素来触发所有可能的状态更改（通常在过程中触发JavaScript事件）。此外，主机组件用语法上合法的值填充输入字段，这些值是通过字段的名称和一组可能的数据范围推断出来的。

在推断出合法的输入值后，FirmFuzz将采用替代策略，将重新请求的相关参数替换为恶意有效载荷

Evaluation

4.1 Firmware Images Tested

3个供应商 6427个镜像 1013个镜像具有基于Linux的文件系统 其中203个推断出网络配置 其中32个镜像有可被FirmFuzz使用的网络接口。

能成功进行Fuzz的设备要大大少于能推断出网络配置的设备，这是因为缺少对特定设备的模拟（例如，系统在启动web服务器之前等待相机可访问）。如果没有可访问的web界面作为入口，FirmFuzz就无法模糊固件映像。

在能够进行Fuzz的镜像中，来自同一设备商的不同设备具有高可复用性的接口。

4.2 Case Study

4.2.1 TRENDnet TEW-673GRU Router

基于MIPS的无线路由器，存在CI漏洞，当用户使用管理员身份登录到设备配置时可被远程利用

漏洞原理(参数未校验)…

FirmFuzz推断出合法输入后，FirmFuzz像管理参数的CGI文件发送POST请求更新参数，而CGI不会校验这些值。

4.3 Comparison with Existing Analysis Frameworks

4.3.1 Firmadyne

Firmadyne测试最初是为特定的嵌入式设备构建的，在不同的设备和不同的供应商中使用相同漏洞的可能性很低。

4.3.2 w3af

即使给它提供了认证必须的信息和修改HTML几个参数，w3af仍然无法使用固件进行身份验证，因此啥都挖不出来

4.3.3 Zed Attack Proxy

ZAP不具有像FirmFuzz那样的身份认证发现能力，为了确保ZAP能正常工作，向它提供了用于管理访问的固件凭据。但ZAP只发现了XSS，这是因为ZAP是黑盒测试，不会像FirmFuzz那样主动与固件交互以获取有关应用程序的上下文。此外，ZAP在测试时不具有监视固件的运行时环境的能力，从而限制它发现漏洞的能力，这些漏洞在被触发时不会通过web应用程序入口点向扫描仪提供任何反馈，即使给它提供了漏洞的PoC。它也还是发现不了。

4.4 Runtime Performance

模糊测试阶段的平均运行时间为16m42s

4.5 Vulnerability Detection Accuracy

现有的扫描仪在测试时会有较高的误报率。它们通常使用暴力方法进行测试，这依赖于服务器的反馈，并且是盲目的。

FirmFuzz对测试中的固件进行运行时监控，且使用上下文输入生成，降低了误报率，且能够定位哪些固件资源存在问题。
