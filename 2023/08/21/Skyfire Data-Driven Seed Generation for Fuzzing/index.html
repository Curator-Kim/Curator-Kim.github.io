Skyfire: Data-Driven Seed Generation for Fuzzing

基于突变的fuzz所产生的大多数输入在早期语法分析阶段被拒绝，基于生成的fuzz所产生的输入可以使模糊处理快速超越语法分析阶段，然而大多数输入无法通过语义检查。

文中提出了一种名为skyfire的新型数据驱动种子生成方法，可以利用大量现有样本中的知识，为处理高结构化输入的模糊程序生成分布良好的种子输入。Skyfire将语料库和语法作为输入，包括两个步骤。Skyfire的第一步是学习概率上下文敏感语法（PCSG），以指定语法特征和语义规则，然后第二步利用学习到的 PCSG 生成种子输入。

Introduction

Skyfire第一步根据语法将收集到的样本（即语料库）解析为抽象语法树（AST），然后学习概率上下文相关语法（PCSG），该语法同时指定了语法特征和语义规则。不同于基于生成的fuzz的上下文无关语法，PCSG 中的每条生成规则都与可应用该生成规则的上下文以及该生成规则在上下文中应用的概率相关联。

在第二步中，Skyfire首先通过在非终端符号上反复选择和应用符合上下文的生成规则来生成种子输入，直到生成的字符串中没有非终端符号为止。在这一过程中，相对于高概率的生产规则，我们更倾向于选择低概率的生产规则，以生成具有不同语法结构的不常见输入。然后，天火会进行种子选择，过滤掉代码覆盖率相同的种子，以减少冗余。最后，天火会根据生产规则随机替换 AST 中的叶级节点，从而对剩余的种子输入进行变异，在保持语法结构的同时，以多样化的方式引入语义变化。

文章主要做了以下贡献：

  • 提出从现有样本中学习概率上下文敏感语法标记(PCSG)，以描述高度结构化输入的语法特征和语义规则
  • 利用PCSG生成具有不同语法结构的种子输入采用种子筛选来减少种子冗余，并通过种子突变来引入不同的语义变化，从而生成多样化且语法语义正确的种子
  • 使用生成的输入对多个 XSLT、XML 和 JavaScript/渲染引擎进行模糊测试，对 Skyfire 进行了评估，结果大大提高了代码覆盖率，并发现了 16 个新漏洞。

Approach Overview

A. Tartget Programs

定义一：一个上下文无关语法(CFG)定义为四元组$G_{cf}=(N,\sum,R,s)$

  • $N$是非终端符号的有限集合
  • $\sum$是终端符号的有限集合

  • $R$是生成规则的有限集合，规则具有如下形式：$\alpha\rarr \beta_1\beta_2\beta_3…\beta_n$，其中$\alpha\in N,n\geq1$，并且$\beta_i\in(N\cup\sum)$，$i=1…n$

  • $s\in N$是特殊的开始符号

B. Overview of Skyfire

Skyfire的三个主要目标

  • 产生合乎语法、合乎语义的种子。
  • 产生多样化的种子
  • 产生不常见的种子，使得输入能够到达程序中less-fuzzed的代码段，并触发潜在的程序行为。

Skyfire的第一步是根据给定语法将样例分割为语法树(ASTs)，并学习概率性上下文敏感语法(probabilistic context-sensitive grammar,PCSG)。PCSG能够同时识别语法特征和语义特征，而CFG只能形容语法特征。具体来说，PCSG 中的每条生成规则都与上下文相关联，在上下文中可以应用该生成规则来表达语义规则。每条生成规则还与该生成规则在给定语境下的应用概率相关联。

在第二部，基于提取的生成规则库，Skyfire首先通过迭代过程生成种子输入，在此过程中，上下文满足要求的生成规则被应用于非终端符号，直到生成的字符串中没有非终端符号为止。在这个过程中，skyfire倾向于低概率的生成规则而非高概率的生成规则，从而生成不同语法结构的非常见输入。然后Skyfire会筛选种子，过滤掉具有相同覆盖率的种子来减少冗余。最后，天火会对剩余的种子输入进行变异，根据生成规则随机替换 AST 中的叶级节点，并将其替换为相同类型的节点，从而在保持语法结构的同时，以多样化的方式引入语义变化。

在接下来两节，将阐述如何学习PCSG来描述语法特征和语义规则（第三节），然后解释如何根据所学习的PCSG生成种子。

Definition 2.

与CFG相似，上下文敏感语法(CSG)是一个四元组$G_{cs}=(N,\sum,R,s)$

  • $N$是非终端符号的有限集合
  • $\sum$是终端符号的有限集合
  • $R$是关于 $[c]\alpha\rarr \beta_1 \beta_2… \beta_n$形式的上下文敏感的生成规则，其中$c$是上下文，形式包括α 的曾祖父母的类型、α 的祖父母的类型、α 的父母的类型、α 的第一个兄弟姐妹的值或 α 的第一个兄弟姐妹的类型（如果值为空），$\alpha\in N, n\geq1$且$\beta_i\in (N\cup\sum)$
  • $s\in N$是特殊的开始符号

论文定义上下文的方式并不是捕捉所有类型的语义规则，而是让大多数语义规则都能被表达出来，并在学习步骤中有效地学习这些规则，在生成步骤中有效地检查这些规则。注意表一中的前五条语义规则可以被捕捉到，但由于上下文建模所需的信息量较大，后两条语义规则无法表达。目前，认为这种分层关系足以捕捉到语义规则的多样性，同时保持较低的冗余度。不过，上下文可以扩展来整合更多规则（例如，通过使用一个以上的兄弟姐妹和/或使用更多或更少的祖先）

Definition 3. (probabilistic context-sensitive grammar (PCSG))

PCSG是一个元组$G_p=(G_{cs},q)$

  • $G_{cs}=(N,\sum,R,s)$是一个CSG

  • q：为每条生成规则分配一个概率，使得对任意$\forall\alpha\in N: $

Seed Generation

基于所学习的PCSG生成种子。

A. Seed Generation

通过最左推导生成种子输入：在生产输入t时，可以先将t设为PCSG的起始符号，然后迭代应用以下步骤直到t中没有非终端符号：

i) 获取 t 中最左侧的非终端符号 $l$ 和相应的上下文 c

ii) 从$R_l$中随机选择一条生产规则$r$，其左侧是给定c的$l$，

iii) 在t中的$l$上应用$r$

这种基于随机最左侧推导的生成方法会在达到时间预算或生成了预定数量的种子输入后终止。由于 PCSG 的性质，生成的大多数输入在语法和语义规则方面都是正确的。注意如果 t 在达到时间预算时仍包含非结束符，则会被丢弃。

Definition 4.

给定PCSG $G_p=((N,\sum,R,s),q)$，最左推导是一串符号序列$t_0,..,t_n$，其中

  • $t_0=s$，即$t_0$仅包含开始符号
  • $t_n\in \sum^$，即$t_n$由终止符号组成，并且$\sum^$表示从$\sum$取出的词序列的所有可能字符串组合
  • 对于$i=1,…,n$，$t_i$从$t_{i-1}$推导而来，通过用R中的生成规则$[c]\alpha\rarr \beta$得到的$\beta$取代$t_{i-1}$中的最左非结束符号$\alpha$（在上下文c下）

由于生成规则选取的随机性，可能会出现一些问题

  • 由于生成规则的递归性，生成可能无法停止。例如XSL中的element可以包含content，content里面也可以包含element。更进一步，当生成规则可以包含许多非终端符号时，问题会被进一步放大，例如一个content包含数百个elements
  • 由于生成规则的递归性，产生的种子可能会没有必要地复杂(例如提深度/广度的复杂性)，带有大量的非终止符号，所产生的结果就是输入会异常巨大，可能会被fuzzer拒绝。

基于启发式最左推导，文章提出了一种生成方法，具体步骤如下算法1所示，与之前的随机性算法的关键区别在于，设计了几种启发式方法来从$R_l$中随机选取规则r

Heuristic 1: 倾向低概率生成规则

根据概率将$R_l$分为高概率规则集$R_l^H$和低概率规则集$R_l^L$，然后根据Heuristic 2和3以高于$R_l^H$的概率(i.e. 0.9)从$R_l^L$中选取规则。注意，分区是通过根据概率对规则进行排序，并找出概率差异最大的两个相邻规则来实现的。

Heuristic2：倾向低频生成规则，并限制同一生成规则的应用次数

记录生成输入时生成规则被应用的频率，对已经应用过 3 次的规则进行裁剪。然后优先选择低频规则，而不是高频规则。

Heuristic3：倾向低复杂度生成规则

为了减少生成输入的不必要的复杂度(广度)，采用规则里含有非终止符号的数量来衡量规则的复杂度，并偏向于低复杂度的规则

Heuristic4：限制规则应用的总数

为了减少生成输入的不必要的复杂度(深度)，限制生成输入时规则应用的总数不超过200

B. Seed Selection

并非所有生成的输入都是唯一和重要的。文章使用代码覆盖率作为选择种子的标准，不依赖模糊器通过耗时的突变和执行程序来被动地保留独一无二的种子，而是在将种子提供给模糊器之前主动选择种子。

具体而言，使用 gcov [29] 来获取开放源代码程序的行覆盖率和函数覆盖率，并使用基于 PIN [30] 的静态分析和覆盖率工具来获取封闭源代码程序的基本块覆盖率。如果一个输入触发了新的代码块覆盖率，就将其作为种子保留下来。

C. Seed Mutation

为进一步确保语法结构的多样性，文章对生成的每个输入进行了小幅突变，即根据生成规则随机替换 AST 中的一个叶级节点，使其成为同类型节点，从而对叶级节点的分布造成临时干扰。换句话说，只有右侧只包含终端符号的生产规则才会被用于这一突变过程。与模糊器中的小步突变（如字节翻转）相比，我们的突变可视为大步突变，是模糊器的小步突变难以实现的。通过随机的位翻转，很难从版本=”1.0 “变为编码=”utf-8”。最后，变异后的输入可以输入模糊器。

B. Vulnerabilities and Bugs Discovered(RQ1)

Bug Results. Skyfire+AFL 发现了 19 个以前未知的内存损坏 bug。但是，Skyfire 只发现了其中的 8 个，而 Crawl+AFL 只发现了其中的 7 个。此外，Skyfire+AFL 还发现了 6 个已知的内存损坏漏洞，但仅使用 Skyfire，只发现了其中一个，而 Crawl+AFL 则发现了其中四个。此外，Skyfire+AFL 还发现了 21 个新的拒绝服务漏洞（包括堆栈耗尽、NULL 指针取消引用和断言失败）。其中，Skyfire 和 Crawl+AFL 只发现了 10 个。请注意，所有错误都是在应用程序执行阶段发现的（见图 1）。总之，Skyfire 和 Crawl+AFL 发现的所有错误也都被 Skyfire+AFL 发现了，而 Skyfire+AFL 发现的错误是 Skyfire 或 Crawl+AFL 没有发现的。这些结果表明，天火可以为模糊器提供良好的种子输入，从而减轻模糊器通过持续执行和突变寻找良好种子的负担，并显著提高模糊器发现处理高结构化输入程序的错误的能力。

Vulnerability Results. 从 19 个之前未知的内存损坏漏洞中，我们成功发现了 16 个新漏洞，其类型如表 V 所示。其中 10 个是越界读/写漏洞，7 个是用户释放后漏洞；共分配了 11 个 CVE 标识，5 份报告仍未解决。至于 Sablotron 中另外三个以前未知的内存损坏漏洞，它们在最新的 Adobe Reader 中无法重现。自 Sablotron 1.0.2 采用以来，Adobe 已对其中两个漏洞进行了内部修复，还有一个漏洞因 Adobe 重构代码而不再可利用。此外，还有 21 个新的拒绝服务漏洞无法利用。对于 Sablotron，由于严重性较低，我们没有向 Adobe 发送错误报告；对于 libxml2，我们在撰写本报告时提交了三份堆栈溢出（即由于无限递归导致的堆栈耗尽）报告。

Capability to Keep Finding Bugs. 图 7 显示了整个模糊测试过程中累计的唯一错误数。X 轴表示模糊测试的天数，Y 轴表示独特错误的累计数量。可以看出，在前八个月中，Skyfire+AFL 和 Crawl+AFL 都持续检测到了唯一错误，但 Skyfire+AFL 比 Crawl+AFL 更有效、更高效地发现了错误。在接下来的 7 个月中，Crawl+AFL 发现独特错误的速度变得极其缓慢。与此不同的是，Skyfire+AFL 仍能持续发现独特的错误，尽管效率越来越低。

C. Code Coverage (RQ2)

表 VI 报告了 Sablotron、libxslt 和 libxml2 在 15 个月模糊测试前后的代码行和函数覆盖率。前三列报告了每个程序的代码行数和函数数。其他列显示了四种方法实现的代码行和函数覆盖率。这里我们只显示 libxml2 2.9.4 的结果，不明确显示 libxml2 2.9.2 和 2.9.3 的结果，因为它们的结果类似。

Overall Coverage Results 在行覆盖率方面，我们抓取的样本覆盖了 Sablotron 的 34.0% 行、libxslt 的 29.6% 行和 libxml2 的 13.5% 行。经过 15 个月的模糊测试后，AFL 的行覆盖率分别提高到 39.0%、38.1% 和 15.3%。通过 AFL，平均有 5.1% 的代码被进一步覆盖。对于 Skyfire，生成的输入覆盖了 Sablotron 的 65.2% 行、libxslt 的 57.4% 行和 libxml2 的 22.0% 行。我们可以看到，生成的输入的覆盖率已经远远高于 Crawl+AFL。经过模糊处理后，AFL 的行覆盖率分别提高到 69.8%、62.5% 和 23.8%，并进一步覆盖了 3.8% 的代码。总体而言，Skyfire+AFL 的行覆盖率比 Crawl+AFL 高出约 20%。

另一方面，在函数覆盖率方面，抓取的样本平均覆盖了 25.2% 的函数；AFL 进一步覆盖了 2.5% 的函数。而Skyfire生成的输入平均覆盖了 41.1% 的函数，AFL 进一步覆盖了 1.7% 的函数。总体而言，Skyfire+AFL 的函数覆盖率比 Crawl+AFL 高出约 15%。

从这些结果可以看出，Skyfire 可以为模糊器提供分布良好的种子输入，从而大大提高模糊器的代码覆盖率。另一个有趣的发现是，在种子输入分布良好的情况下，通过 AFL 提高的覆盖率反而降低了，即行覆盖率从 Crawl+AFL 的 5.1% 降至 Skyfire+AFL 的 3.8%，函数覆盖率从 Crawl+AFL 的 2.5% 降至 Skyfire+AFL 的 1.7%。这可以解释为，Skyfire减轻了模糊器寻找种子的负担，使模糊器可以专注于在这些有趣的部分触发意外行为。

Detailed Coverage Results.

我们从源代码文件层面详细分析了 Sablotron、libxslt 和 libxml2 的行覆盖率和函数覆盖率。表 VII、VIII 和 IX 报告了所有 .c 或 .cpp 文件的覆盖率。第一列显示了文件名，第二列列出了在文件中发现的唯一错误数。与表 VI 类似，接下来的四栏列出了每个文件的行覆盖率，最后四栏列出了每个文件的函数覆盖率。

从这些表格中我们可以看到，与 Crawl+AFL 相比，Skyfire+AFL 提高了 Sablotron 25 个文件中的 21 个、libxslt 32 个文件中的 26 个以及 libxml2 38 个文件中的 28 个的覆盖率。这表明大多数文件的覆盖率都得到了提高。此外，我们还手动查看了代码覆盖率极低的文件，并分析了覆盖率如此之低的原因。首先，有些文件是用于测试的，因此不会被执行（例如 libxslt 中的 testThreads，以及 libxml2 中的 runsuite、runtest 和 runxmlconf）。其次，有些文件只有在特定配置下才能执行。例如，我们通过命令行运行 libxslt，但只有通过 Python 接口运行 libxslt 时，才能覆盖 libxslt-py、libxslt 和 types。同样，我们使用默认的命令行配置运行 libxml2，因此没有覆盖 libxml2 中的许多功能（如 xmlcatalog 和 c14n），整体覆盖率较低。不过，可以在模糊测试前通过设置各种命令行配置来触及这些功能。第三，有些文件包含处理已废弃功能的代码（如 Sablotron 中的 sdom 和 libxml2 中的 SAX），因此不会被执行。

Code Coverage vs. Bugs.

我们还分析了文件的代码覆盖率与该文件中发现的错误数量之间的关系，如图 8 所示。我们可以看到，所有错误都是在行覆盖率超过 45% 和函数覆盖率超过 55% 的文件中发现的。这表明，提高代码覆盖率有助于发现漏洞，fuzzer可以利用覆盖率信息来指导模糊过程，从而专注于覆盖率较低的程序文件。

D. Effectiveness of Context and Heuristics (RQ3)

Context. 为了评估上下文在 PCSG 中的有效性，我们实施了一种基于 CFG 的输入生成方法，该方法使用启发式 3 和 4 来解决非终止问题（见第四-A 节），并比较了所生成的输入中通过语义检查阶段的百分比（见图 1）。为了进行公平比较，我们使用这两种方法为 XSL 和 XML 生成了相同数量的输入（即 10,000 个）。请注意，通过查看退出代码，我们可以轻松确定输入是否未通过语义检查。

结果显示，基于 CFG 方法生成的 XSL 输入没有一个能通过语义检查，因为它们都没有通过表 I 中的前两条语义规则。同样，基于 CFG 方法生成的 XML 输入只有 34% 可以通过语义检查。相反，通过考虑上下文，Skyfire 生成的 XSL 和 XML 输入中分别有 85% 和 63% 通过了语义检查并进入了应用执行阶段。这表明，考虑到上下文，PCSG 在生成可通过语义检查的输入方面比 CFG 更有效。

Heuristics. 我们在第 V-B 和 V-C 节中的评估表明，天火可以生成分布良好的种子，这也反映了启发式算法 1 和 2 的有效性（见第 IV-A 节）。为了评估启发式方法 3 和 4 的有效性，我们测量了天火的终止率，即在特定数量的生产规则应用中生成的输入数量。在这里，我们将生产规则应用的总数从 20 到 400 不等，并生成了 1,000 个输入。如图 9a 所示，在不使用启发式方法的情况下，100 次生产规则应用中只生成了 20% 的 XSL 输入；而在使用启发式方法的情况下，这一终止率上升到了 50%。至于 XML 输入，在不使用启发式方法和使用启发式方法的情况下，在 100 次生产规则应用中分别有 55% 和 65% 的输入生成。这表明启发式方法 3 和 4 有效地解决了非终止问题。请注意，在我们的实施过程中，我们根据经验将生产规则应用的总次数（见启发式 4）设为 200 次，以便根据图 9a 在终止和复杂性之间取得平衡。

分别生成 20 万个 XSL 和 XML 输入，并计算了生成一个输入所需的规则应用次数分布。如图 9b 所示，我们可以看到大多数输入都是在 45 次生产规则应用内生成的，这反映出生成的输入大多并不复杂。这表明启发式 3 和 4 有效地减少了种子生成的不必要复杂性。

E. Performance Overhead(RQ4)

评估Skyfire的效率：测量了从抓取的样本（18,686、19,324 和 525,647 个 XSL、XML 和 JavaScript 样本）中学习 PCSG 的执行时间，以及生成 18,686、19,324 和 525,647 个 XSL、XML 和 JavaScript 种子的执行时间。如表 X 所示，XSL 和 XML 的 PCSG 学习时间约为 1.6 小时，其生成步骤仅耗时约 21 秒。由于样本数量较多，JavaScript 的 PCSG 学习时间为 41.0 小时。我们可以看到，学习和生成步骤的计算时间与抓取的样本数和生成的种子数呈线性关系，具有可扩展性。

F. Case Study and Discussion

从表 IV 中我们可以看到，天火可以生成直接触发漏洞的输入。例如，图 4a 就是由天火生成的，它触发了 Sablotron 1.0.3 中的缓冲区下溢漏洞。然而，有些错误并不能通过我们生成的输入来发现，而是需要利用模糊器的能力来检测。例如，图 10a 中的代码片段由 Skyfire 生成，没有触发任何错误。然而，当它被输入到 AFL，并被 AFL 变更成图 10b 所示的代码片段时，Sablotron 中的越界访问漏洞被触发，该漏洞也存在于最新的 Adobe Reader 和 Adobe Reader DC 的 XSLT 引擎中。

特别是，当 valueof 元素提供的 XPath 表达式对 substring-after() 或 substring-before() 的调用不正确时，就会触发该漏洞。当漏洞触发时，未初始化的 Expression Atom 对象可通过越界读取获取，然后用于后续对 tostring() 的虚拟函数调用，只要攻击者控制了未初始化的内存内容，就可以直接进行远程利用。

如图 11 所示，该漏洞的根本原因是 expr.cpp 中对 substring-before() 和 substring-after() 的检查不正确，即第 941 行的参数计数检查是在使用参数的两个 tostring() 调用（第 939 和 940 行）之后进行的。因此，当只提供一个参数时，如图 10b 所示，atoms[1] 是一个未初始化的 Expression 原子指针，会导致通过 tostring() 调用执行代码。

在这个例子中，AFL 的突变删除了函数调用 substring-before 的第二个参数，而这是 Skyfire 的生成和突变无法实现的。这是因为天火涉及大步突变（如替换属性），而 AFL 采用小步突变（如字节翻转）。这表明，天火是模糊器的良好补充。通过将它们结合在一起，我们可以大大提高错误查找能力。

G. Preliminary Results for JavaScript

我们使用变量名规范化后生成的 JavaScript 种子样本，结合相对固定的 HTML 模板，对 Internet Explorer 11 的 Trident 渲染引擎（mshtml.dll）进行了为期约两个月的模糊测试。结果表明，作为一种比 XML 和 XSL 更复杂的语言，JavaScript 有更多的语义规则。另一方面，Internet Explorer 的 JavaScript 和渲染引擎通常会操作大量对象，这些对象之间的交互方式更为复杂，代码覆盖率可能不是描述此类交互的良好指标。因此，需要一个更好的指标来捕捉此类交互，并在模糊测试过程中用作反馈。

JavaScript 本身的设计和实现复杂性还带来了其他一些微妙的问题。其中一个例子是变量及其作用域，要生成真正语义有效的 JavaScript 代码段，就必须对包含变量定义、赋值和引用等语义规则的现有代码段进行单独解析和注释；将这些代码段链在一起时，变量名可能会被规范化，变量的初始化和引用可能会在尊重这些语义规则的情况下发生演化交叉。

我们今后的工作是继续扩展我们的方法，以便更好地支持 JavaScript 和 SQL 等更复杂的语言。具体来说，对于 JavaScript，我们计划以 Webkit JavaScriptCore 为目标，它是开源的，而且更加独立；对于 SQL 语言，我们计划对 sqlite3 进行模糊处理。

从数量上看，在撰写本文时，我们已经在 Internet Explorer 11 上运行了 1,591,263 次 JavaScript 种子输入，发现了 11 个 NULL 指针解除引用 bug，并实现了 PIN [30] 测量的 85% 的基本区块覆盖率。不过，这些结果仍然证明了天火的种子生成能力很有前途。在今后的工作中，我们将继续充实天火的策略，以便对 Internet Explorer（或其他浏览器）进行模糊测试。
