NAUTILUS: Fishing for Deep Bugs with Grammars

项目地址：https://github.com/RUB-SysSec/nautilus

NAUTILUS: 通过结合语法使用和代码覆盖率来有效进行模糊测试的方法。这使得Interesting Inputs可以被有效利用，从而提升输入能够通过语法和语义检查的可能性。论文实现了一种概念验证(Proof of Concept)的模糊测试器，并测试了多个目标包括ChakraCore, PHP, mruby和lua。NAUTILUS发现了大量的漏洞：mruby7个，PHP3个，ChakraCore2个和Lua1个。总共授予了2600个USD和6个CVE

I. Introduction

Goals and contributions. 将基于形容的输入生成和基于反馈的模糊测试结合来探索程序深层漏洞。NAUTILUS并不需要语料库，只需要应用程序的源码和产生输入的语法。测试进程可以通过删除部分无趣的语法或假如额外的规则来进一步调整，例如从语言文档中整合函数名称和参数类型。另外，NAUTILUS运行用户通过额外脚本扩展语法，从而令NAUTILUS产生更多可选输入语言来增强产生语义合法输入的能力。

NAUTILUS并不只是产生有趣的输入，模糊测试本身也会利用语法，对输入进行高级语义转换，例如，将程序中的一个表达式换成另一个表达式。通过将这些突变与覆盖率反馈相结合，NAUTILUS 可以创建一个语义有趣、种类繁多的输入语料库，并将它们重新组合，从而大幅提高找到语法和语义均有效的新输入的概率。

总而言之，作了以下贡献：

  • 介绍并评估了NAUTILUS，它是第一款将基于语法的输入生成与反馈导向模糊处理相结合的模糊器，大大提高了对需要高结构化输入的目标进行模糊处理的效率和效果，而无需任何语料库，并且支持图灵完备脚本，从而创建对复杂、无上下文输入语言的描述
  • 描述并评估了几种基于语法的突变、最小化和生成技术。通过将覆盖率反馈与基于语法的拼接相结合，NAUTILUS 能够生成语法上正确、语义上通常也正确的程序，其性能优于传统的纯生成式模糊器，后者需要花费大量时间生成和测试语义无效的输入。
  • 发现了几个漏洞

II. Background

A. Fuzzing

B.Context-Free Grammars

Context-free Grammars (CFGs)非常适合用户形容高结构化语言，这里给出一个CFG简单的定义：直观地说，CFG 是一组形式为 “某个变量 X（非终止符号）可以被下面的字符串（终止符号）和变量数组替换 “的生成规则。此外，一个特殊的起始非终止符号还指定了应用这些规则的起始位置。CFG 所描述的输入语言是所有字符串的集合，这些字符串可以通过应用任意数量的规则得到，直到不再出现非终止符号为止。

CFG正式定义：$G = (N, T, R,S)$，其中

  • $R$是非终止符号的有限集。非终止符号可以被认为是语言中间状态的形容。
  • $T$是终止符号的有限集，$N$和$T$不相交
  • $R$是包含生成规则的有限集，形式为$A\rarr \alpha$，其中$A\in N$且$\alpha\in ^*T\cup N$
  • $S\in N$是开始符号(非终止符号)，所有CFG生成词应从$S$开始

由于每条规则左侧只包含一个非终止符号，因此可能的生成只取决于一个非终止符号，而无需上下文，因此这些语法被称为无上下文语法。

要推导出一个字符串，必须对起始符号 S 应用匹配的生成规则，即左侧包含相应非终止符的规则。每一步都会有一个非终端被匹配该非终端的规则的右侧所替换。

Example II.1 考虑以下语法

一个可能的G1生成是：$PROG\rarr STMT \rarr VAR=EXPR\rarr a=EXPR \rarr a= NUMBER \rarr a=1$

最终产生的字符串即a=1

CFG生成的每个字符串都可用派生树来表示。NAUTILUS 主要对这些派生树进行操作，而不是对树的字符串表示（我们称之为未解析字符串）进行操作。派生树是 NAUTILUS 的内部表示，用于进行结构突变。由于大部分语言并非上下文无关(如校验和或XML：开头和结尾需要相同的标记符)，论文扩展了CFG，允许使用额外的脚本来转换输入。

III. CHALLENGES

四个关键方面：

  • C1: 语法和语义有效的输入生成
  • C2: 独立于语料库。大部分Fuzzer需要一个初始输入语料库，但要对程序中新的困难的部分获取语料库则比较困难，尤其是内部或未发布的软件，因此fuzz一个应用程序时最好不需要语料库。不过，可以利用输入规范。此外，试图将种子语料库与输入规范结合起来的技术通常需要限制其输入语言，以便能够进行解析，因此，在使用输入规范时，种子语料库的作用会降低。
  • C3: 高覆盖率。这主要涉及两个方面：(1) 通过输入处理；(2) 引导分析以探索新路径。
  • C4: 表现优秀。快速测试周期意味着每个模糊窗口要测试大量输入，这是提高模糊器有效性的关键因素之一。为确保高执行率，输入量要小，生成方法要快。

纯语法输入推导可解决C1和C2问题，我们使用反馈驱动输入生成来解决C3问题，而引导输入推导长度以及最小化有趣输入可解决C4问题。

IV. Design of NAUTILUS

如图1所示，第一步是使用插桩对源码进行编译，从而获取运行时覆盖率反馈信息。第二步，启动fuzzer进行对用户提供的语法进行解析；第三步，产生少量(1000个)初始化输入并传递给调度程序；第四步和第五步，NAUTILUS会测试所产生的输入是否触发新路径，如果是则将其最小化并加入种子队列；第六步和第七步，基于是否仍可以探索新路径，NAUTILUS选择对已有输入进行变异或重新产生新输入；第八步和第九步，突变方法包括用新生成的树替换子树或将发现目标应用不同有趣特征的树组合在一起等技术，在突变完成后将输入添加到队列中用于后续分析运行。

A. Generation

生成算法应该应用到语法的不同方面，从而可以最大化覆盖率。论文的模糊器内部只使用树表示法，而不是单词的未解析字符串表示法。这允许我们对树进行操作，并自定义解析例程。论文使用语法规则从集合推导树中推导出随机树。如果一个非终端有多个规则，那么就有多种方法来选择一个规则进行进一步推导。这里评估了两种方法：naive generation和uniform generation

naive generation就是随机选择一个可应用的规则，但这会导致生成大量相似的输入，因此应用了一个筛选器检查输入最近是否已被产生过。

Example IV.1 (Naive Generation). 在Example II.1中，从语法 G1 生成输入树时，STMT 非终止有两条规则。如果选择规则 3，生成立即终止。如果选择规则 4，则生成继续，并且可以生成多棵不同的树。如果这两条规则都以 50%的概率被选中，则生成的输入中有一半是返回 1。

Uniform generation则充分考虑了语法结构带来的强偏差，能够从所有可能的树集中统一抽取树。该算法接受一个 CFG 和一个整数 n ≥ 0，并返回长度为 n 的字符串，这些字符串从给定的输入语法中统一导出。该算法根据规则能生成的不同子树的数量来选择规则：对于每个非终止 n、每个可能的长度 l、n 的每个生产规则 r，都会计算出可能的子树数量 p(n,l,r)。举例来说，一条能生成四个不同子树的规则被选中的次数是一条能生成两个子树的规则的两倍。

B. Minimization

在找到interesting inputs后，NAUTILUS会尝试触发相同覆盖率情况下，生成更小的输入，这样能够有更短的执行时间，以及潜在变异位置的更小集合。采用两种方法来最小化输入：

  • Subtree Minimization. 目的是在触发新路径转换的同时，尽可能缩短子树。对于每个非终端，我们都会生成尽可能小的子树。然后，我们用该位置上可能的最小子树依次替换每个节点的子树，并检查新的转换是否仍由更改后的输入触发。如果过渡仍被触发，则用改变后的输入取代原来的输入，否则丢弃改变后的输入（见例 IV.2）。

  • Recursive Minimization. 该策略在子树最小化之后使用。其目的是通过识别递归并逐个替换来减少递归的数量。

C. Mutation

非指定情况下平均选择以下策略

  • Random Mutation. 从一棵树中随机选取一个节点，并用随机生成的、根植于同一非终端的新子树替换它。子树的大小是随机选择的，子树的最大大小是一个配置参数。

  • Rules Mutation. 用所有其他可能的规则生成的子树顺序替换输入树的每个节点，这种突变类似于 AFL 使用的确定性阶段。

  • Random Recursive Mutation. 随机选取一颗递归树并重复$2^n(1\leq n\leq 15)$次，从而生成嵌套度更高的树。

  • Splicing Mutation. 通过从一个interesting input中提取一棵子树并将其放入另一个输入中，将找到不同路径的输入组合在一起：用队列中另一棵树的 “合适 “子树替换其中一棵子树。为此，它会随机选取一个内部节点，作为要替换的子树的根节点。然后，它从队列中的一棵树中随机抽取一棵根植于相同非终端的子树来替换旧的子树。

  • AFL Mutation. 采用AFL中使用的变异措施，这里对字符串进行操作，所以子树在变异前需要转换为字符串。变异后可能产生不合法的树，从而找到parser的bugs。变异主要包含比特翻转、数学运算和Interesting values替换等

    之后，经过变异的终端字符串会被存储为一个新的自定义规则，并以相同的起源非终端添加到语法树中，取代原来的子树。自定义规则不会添加到语法中，而是与语法树一起保存在本地。例 IV.5 展示了这一过程。

D. Unparsing

在NAUTILUS获得候选推导树后，NAUTILUS需要从中生成实际输入。将推导树转化为二进制文件的这一步通常称为解解析。对于真正的 CFG，这一过程非常简单。解解析函数是递归定义的，用于连接所有未解析的子树。然而，如前所述，现实世界中的许多输入语法并不是上下文无关的。因此，我们通过一个额外的解解析脚本来扩展规则定义，该脚本可以对所有未解析子树的结果进行任意计算。脚本支持是我们生成式方法的一大优势，因为我们无需解析输入。因此，我们可以自由地超越可判定语法，而不会有任何缺点。Skyfire [49] 或 IFuzzer [47] 等方法仅限于支持解析的语法。例 IV.6 展示了这种技术如何生成语法和语义都有效的 XML 文档。

V. Implementation

采用Rust实现NAUTILUS，整体架构类似于AFL，采用mruby解释器来执行可嵌入扩展语法的脚本。与AFL类似，NAUTILUS也要求目标程序必须是插桩的。它在多个阶段对目标程序进行模糊处理。下文将详细介绍这些过程。

A. Target Application Instrumentation

与AFL相同，NAUTILUS采用64KB的位图来记录覆盖率信息，通过forkserver提高运行速度

B. ANTLR Parser

NAUTILUS接受的语法输入既可以是JavaScript Object Notation（JSON）（NAUTILUS使用的自然语法表示法），也可以是为ANother Tool for Language Recognition（ANTLR）[1]编写的语法，因为已有200多个适用于多种编程语言的ANTLR语法公开发布[6]。为了支持ANTLR语法，论文集成了一个ANTLR解析器组件，可将ANTLR语法转换为NAUTILUS的本地格式。该解析器可自动转换大多数语法。在某些情况下，ANTLR 语法不指定空白，因为空白在解析过程中无关紧要；但是，空白在输入生成过程中却很重要。因此，通常需要在这些语法的一些关键规则中添加空格。

C. Preparation Phase

NAUTILUS会基于语法预计算一些数据：

  • 对于每个非终端 n，生成以 n 为起始非终端的字符串所需的最小规则数。这些数据将用于规则突变（第 IV-C 节）。
  • $p(n,l,r)$. 对于每个非终端 n，对于每个可能的长度 l，对于 n 的每个生产规则 r，以 r 为第一规则，应用 l 个规则，可能的以 n 为根的子树的数目。Uniform Generation（第 IV-A 节）需要这些信息。
  • $p(n,l)$. 对于每个非终端 n，对于每个可能长度 l，可能子树的数量。这表示从非终端 n 开始应用 l 条规则可以生成多少可能的子树，或者换句话说，有根 n 和 l 条边的衍生树的数量。Uniform Generation（第 IV-A 节）也需要这些信息。

计算非终止最短长度的算法类似于Purdom所提出的，其他计算算法采用了McKenzie所提出的

D. Fuzzing Phase

图二展示了NAUTILUS在模糊测试阶段的工作流程图，在产生一些初始输入后，调度器决定哪些应该首先被测试：(i) 对已存在输入进行足够变异 (ii) 从头开始产生新输入。调度器控制着一个队列，该队列包含所有生成或变异的输入，这些输入被认为是有趣的，也就是说，它们中的每一个都会触发应用程序中至少一个基本块之间的转换，而其他输入不会触发。

调度程序会按顺序处理队列中的每个项目。队列中的每个项目都有一个状态，表示从队列中取出后将如何处理。状态可以是以下值之一：

  • init. 当一个输入触发了新转换时，它会被保存到队列中，并标为init状态。当调度器选择一个对象是init状态时，它会被minimized，在完成最小化后，状态被设置为det
  • det. 使用Rules Mutator、Random (Recursive) Mutator和Splice Mutator对 det 中的项目进行突变（见第 IV-C 节）。当规则突变器完成对某个项目的突变后，由于不可能再进行该类型的突变，该项目将转入 detafl 状态。
  • detafl. 使用 AFL 变异器、Random (Recursive) Mutator和Splice Mutator对 det 中的项目进行变异。当 AFL 变异器处理完一个项目后，它就会进入random状态。
  • random. 这是每个条目的最终状态。只有Random (Recursive) Mutator和Splice Mutator会应用到这个条目上。与AFL不同的是，我们不会在完成每个阶段后再继续下一个输入。相反，NAUTILUS 只在每个输入上花费很短的时间（通常为几秒钟），然后再继续下一个输入。因此，我们可以快速探索那些极有可能产生新覆盖范围的输入，而不会在非生产性输入上花费过多时间。这样，我们就能达到类似 AFLFast [25] 的效果。

输入树被选中执行后，会被解解析为输入字符串。然后，使用与 AFL 类似的 fork 服务器运行目标程序，从而高效地启动目标程序。接下来可能会出现三种状态 (i) 目标程序在执行过程中崩溃，那么，导致崩溃的输入的二进制表示将被保存在一个单独的文件夹中；(ii) 输入导致目标程序采取了新的路径，那么，输入的树形表示将被添加到队列中；或者 (iii) 输入没有触发任何新的转换，输入将被丢弃。

C. Evaluation Against Other State-of-the-art Fuzzers

采用分支覆盖率作为评价标准，与用于反馈导向模糊的AFL和基于语法模糊的IFuzzer进行比较

为了避免不同fuzzer内部记录覆盖率方法带来的差异，使用GCOV来测量分支覆盖率

基线表示在没有与模糊器交互的情况下，由我们生成的语料库本身发现的覆盖率。NAUTILUS发现的额外覆盖率要高得多：AFL和IFuzzer在ChakraCore、mruby和PHP上发现的额外覆盖率在0到1.9个百分点之间，而NAUTILUS发现的额外覆盖率在10.0到28.1个百分点之间。对于 Lua，AFL 发现了 15.2 个额外百分点，而 NAUTILUS 发现了 27.2 个额外百分点。由于我们为 AFL 提供了良好的字典和大量的案例库供其学习，因此我们认为这一设置是一个很强的基准线，也是需要达到的一个重要标准。尽管 IFuzzer 基于良好的语法，但其性能几乎没有超过 AFL，这一事实也证明了这一点。

我们按照 Arcui 等人[21]的建议进行了 Mann-Whitney U 检验，以确保统计显著性，结果见表 III。在所有情况下，我们的最差运行结果都优于所有其他工具的最佳运行结果。因此，U 检验得出的 p 值非常小，我们可以排除观察到的差异是随机机会造成的可能性。所有统计数据均使用 Python scipy [38] 和 numpy [40] 库计算。

针对基于语法的输入生成（问题3）和反馈导向模糊（问题4）的相对优势，我们禁用了NAUTILUS中的覆盖反馈机制，并在与完整版相同的环境中运行。该实验旨在证明，在其他条件（实现和语法的性能）相同的情况下，将反馈和语法模糊相结合确实会产生显著的性能优势。我们将本实验视为评估 Peach [10] 或 Sulley [17] 等模糊器的替代物。我们无法直接评估这些工具，因为它们需要手动编写生成器而不是语法。

结果见图 4 和表 II（标注为无反馈）。我们发现，在所有情况下，使用纯粹的生成语法模糊器都比 AFL 的覆盖率有显著提高。

不出所料，盲语法模糊比 AFL 的突变模糊效果更好（在 mruby 的情况下，提高了一个数量级以上），即使给出大量种子也是如此。这表明，即使在 AFL 和相关工具发布之后，语法模糊仍然具有重要意义（问题 3）。然而，将反馈添加到语法模糊方法中会带来更大的改进： 在所有情况下，我们发现的新分支数量都是盲语法模糊器的两倍以上。因此，我们得出结论：在语法模糊中添加反馈是提高模糊性能的一个非常重要的步骤（问题 4）。

Case Study：Feedback Grammar Fuzzing. 在对 mruby 进行模糊测试时，NAUTILUS 自动学习了以下有趣的代码片段： ObjectSpace.each{|a| a.method(…) } 它允许模糊器测试存在的任何对象的方法调用。这大大提高了找到方法的正确接收器的几率。只要模糊器猜测到一个正确的方法名称，该构造就会立即产生新的覆盖范围。然后，模糊器就可以逐步学习如何为这一特定调用构造有效参数。最后，如果模糊器需要特定的接收器对象，它只需在输入的某个地方创建该对象，因为任何对象都能接收方法调用。AFL 风格的模糊器无法利用这一点。即使以字典形式给出了一组有效的方法名称，它也无法为调用构造复杂的参数。盲语法模糊器无法可靠地生成这个包含有趣调用的小工具。NAUTILUS 利用这个小工具发现了两个 CVE，如果没有适当的语法，使用盲语法模糊或 AFL 式模糊都很难发现这两个 CVE。

在使用重要符号扩展语法时，我们手动将字符串工具的输出添加到语法中。这一步骤很容易实现自动化，从而进一步减少所需的人工工作量。

此外，从程序本身自动推断语法的技术，如 Höschel 等人提出的工具[37]或 Bastani 等人提出的工具[23]，或使用机器学习技术[33]，也可以大大减少对语法的依赖。
